# Forge — Экономическая модель

## Вычислительный стандарт (Compute Standard)

Каждая денежная система подкреплена редкостью:

| Эпоха | Стандарт | Обеспечение |
|-----|----------|---------|
| Античность | Золото/Серебро | Геологическая редкость |
| 1944–1971 | Бреттон-Вудс | USD, привязанный к золоту |
| 1971–сегодня | Петродоллар | Спрос на нефть + военная мощь |
| 2009–сегодня | Bitcoin | Электричество, сожженное на SHA-256 |
| **Forge** | **Вычислительный стандарт** | **Электричество, потраченное на полезный инференс** |

Forge вводит Вычислительный стандарт: единица стоимости подкреплена реальными затратами энергии на выполнение полезных вычислений. В отличие от Proof of Work в Bitcoin, каждый джоуль, потраченный в Forge, производит реальный интеллект.

## CU: Нативная валюта

### Что такое CU

**1 CU = 1 миллиард FLOP (операций с плавающей точкой) верифицированной работы по инференсу.**

CU — это не криптовалюта. Это не токен в блокчейне. Это расчетная единица, представляющая выполненные реальные вычисления. CU имеет ценность, потому что это право на будущие вычисления: если вы заработали CU, обслуживая инференс, вы можете потратить их на получение инференса.

### Почему CU, а не Bitcoin

| Свойство | CU | Bitcoin |
|----------|-----|---------|
| **Обеспечение стоимости** | Полезные вычисления (внутреннее) | Хеш-вычисления (искусственное) |
| **Скорость расчета** | Мгновенно (локальный леджер) | От секунд до минут (Lightning/цепь) |
| **Стоимость транзакции** | Ноль | Комиссии каналов, комиссии сети |
| **Внешняя зависимость** | Нет | Состояние сети Bitcoin |
| **Квантовый риск** | Нет (нет криптограф. головоломки) | Уязвимость SHA-256 / ECDSA |
| **Генерация дохода** | Да (простаивающее железо приносит CU) | Нет (BTC в кошельке ничего не приносит) |

CU является **основной** расчетной единицей. Bitcoin, стейблкоины и фиат — это опциональные **выходы (off-ramps)**, доступные через адаптеры мостов вне протокола.

### CU как продуктивный актив

```
Многоквартирный дом          Mac Mini на Forge
───────────────────         ──────────────────
Актив: здание               Актив: оборудование
Затраты: обслуживание       Затраты: электричество
Доход: аренда               Доход: CU от инференса
Прибыль: аренда - обслуж.   Прибыль: заработанные CU - электр.
Простой = потеря дохода     Простой = упущенный потенциал
```

Вычислительное устройство в Forge не похоже на Bitcoin в кошельке (статичная стоимость, нет дохода). Оно подобно арендной недвижимости — генерирует доход через полезную работу.

## Модель транзакций

### Выполнение сделки

Каждый инференс создает сделку между двумя сторонами:

```rust
pub struct TradeRecord {
    pub provider: NodeId,       // Кто выполнял инференс
    pub consumer: NodeId,       // Кто запрашивал
    pub cu_amount: u64,         // Передано CU
    pub tokens_processed: u64,  // Выполненная работа
    pub timestamp: u64,
    pub model_id: String,
}
```

Сделка записывается обеими сторонами. В текущей реализации каждый узел ведет локальный леджер. В целевой реализации добавятся двойные подписи и синхронизация через gossip.

### Динамическое ценообразование

Цены в CU колеблются на основе местного спроса и предложения:

```
effective_price = base_cu_per_token × demand_factor / supply_factor
```

- **Больше свободных узлов** → supply_factor растет → цена падает
- **Больше запросов на инференс** → demand_factor растет → цена растет
- Каждый узел наблюдает за своими рыночными условиями. Глобального стакана ордеров нет.

### Бесплатный уровень

Новые узлы без истории вклада получают 1000 CU. Это позволяет любому сразу начать пользоваться сетью. Бесплатный уровень расходуется с первым запросом и не восстанавливается.

Защита от атак Сивиллы: если в леджере появилось более 100 неизвестных узлов без вклада, новые запросы по бесплатному уровню отклоняются.

## Доказательство полезной работы (Proof of Useful Work)

### Концепция

Proof of Work в Bitcoin: "Я сжег электричество, вычисляя хеши SHA-256. Вот нонс, который это доказывает".

Proof of Useful Work в Forge: "Я сжег электричество, выполняя инференс LLM. Вот ответ, а вот подпись потребителя, подтверждающая, что он его получил".

Ключевое отличие: доказательство Bitcoin генерируется самостоятельно (любой майнер может создать валидный хеш). Доказательство Forge требует **контрагента** — того, кому действительно был нужен инференс. Спрос невозможно подделать.

### Протокол верификации (целевой)

```
1. Потребитель отправляет InferenceRequest провайдеру
2. Провайдер выполняет инференс, стримит токены обратно
3. Потребитель получает токены, вычисляет хеш ответа
4. Обе стороны подписывают TradeRecord:
   - Провайдер подписывает: "Я это вычислил"
   - Потребитель подписывает: "Я это получил"
5. TradeRecord с двойной подписью рассылается по сети (gossip)
6. Любой узел может проверить обе подписи
```

Узел не может раздуть свой баланс CU без сотрудничающего контрагента. Сговор возможен, но экономически иррационален — сговаривающийся потребитель ничего не выигрывает, подписывая фальшивые сделки.

### Текущая реализация

Текущая эталонная реализация использует локальные леджеры с защитой целостности HMAC-SHA256. Двойные подписи и gossip — следующий шаг.

## Доход и репутация

### Доход (Yield)

Узлы, которые остаются в сети и предоставляют вычисления, зарабатывают доход:

```
yield_cu = contributed_cu × 0.001 × reputation × uptime_hours
```

При репутации 1.0 узел, внесший 10 000 CU, зарабатывает 80 CU за 8-часовую ночь. Это не инфляция, а награда за доступность. Узлы, которые надежно находятся в сети, более ценны для системы.

### Репутация

Каждый узел имеет показатель репутации от 0.0 до 1.0:

- Новые узлы начинают с 0.5
- Время в сети и успешные сделки повышают репутацию
- Отключения и неудачные сделки снижают репутацию
- Выше репутация → выше ставка дохода, приоритет при планировании задач

## Расчеты и внешние мосты

### Основное правило

**Протокол ведет расчеты в CU.** Конвертация во что-либо другое — это вопрос интеграции.

### Отчеты о расчетах

Операторы могут экспортировать аудируемую историю сделок за любой промежуток времени:

```
forge settle --hours 24 --price 0.05 --out settlement.json
```

Отчет включает: валовый заработок в CU, валовые траты в CU, чистые CU, количество сделок и опциональную справочную цену за CU.

### Архитектура моста

```
Уровень 0: Протокол Forge
  → Учет CU, сделки, цены

Уровень 1: Отчет о расчетах
  → Экспортируемая история сделок
  → Справочный обменный курс

Уровень 2: Внешний мост (опционально)
  → CU ↔ BTC (Lightning)
  → CU ↔ stablecoin
  → CU ↔ фиат
```

Слой моста находится вне протокола. Разные операторы могут использовать разные мосты. Протокол остается полезным даже при нулевой внешней ликвидности.

### Мост Lightning

Для операторов, которым нужны расчеты в Bitcoin:

```bash
forge settle --hours 24 --pay
```

Это создает Lightning-инвойс (BOLT11) на сумму чистых заработанных CU, сконвертированных по заданному курсу (по умолчанию: 10 мсатоши за 1 CU).

## Бюджеты под управлением агентов

### Видение

Традиционно: Человек решает → Человек платит → ИИ выполняет
Forge: Политика разрешает агенту → Агент проверяет бюджет → Агент тратит CU → Агент выполняет

### API

```
GET /v1/forge/balance   → баланс CU, вклад, потребление, репутация
GET /v1/forge/pricing   → рыночная цена, оценка стоимости за 100/1000 токенов
```

Агент может:
1. Проверить свой баланс перед отправкой запроса
2. Оценить стоимость инференса по текущим рыночным ценам
3. Решить, стоит ли запрос затрат в CU
4. Выполнить и оплатить автоматически

Люди-кураторы устанавливают бюджетные политики. Агенты работают в этих рамках автономно.

### Цикл самосиления

```
Агент (малый, на телефоне)
  → зарабатывает CU, одалживая свободные мощности
  → тратит CU на доступ к более крупной модели
  → становится умнее
  → принимает лучшие экономические решения
  → зарабатывает больше CU
  → получает доступ к еще более крупным моделям
  → ...
```

Это возможный паттерн применения. Протокол предоставляет рынок; агенты — стратегию.

## Почему это не Web3

Большинство проектов Web3 создают искусственную редкость (токены) поверх избыточных цифровых благ. Forge делает наоборот:

- **Вычисления действительно редки** — они требуют реального электричества, реального кремния, реального времени
- **CU не спекулятивен** — он представляет верифицированную работу, а не ставку на будущее признание
- **Никаких ICO, никаких продаж токенов, никаких токенов управления** — CU зарабатываются трудом
- **Блокчейн не требуется** — двусторонних подписей и gossip достаточно
- **Никаких смарт-контрактов** — протокол сам является контрактом

Стоимость создается физикой, а не консенсусом.
