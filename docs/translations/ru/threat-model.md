# Forge — Модель угроз

## Цели безопасности

1. **Конфиденциальность при передаче**: пассивные наблюдатели и реле не должны иметь возможности читать промпты или ответы.
2. **Аутентифицированные пиры**: каждое прямое соединение должно быть привязано к криптографическому идентификатору узла.
3. **Ограниченное доверие**: система должна явно указывать, какие узлы видят открытый текст, а какие — только промежуточные состояния.
4. **Доступность**: сбои отдельных пиров должны приводить к деградации сервиса, а не к незаметному повреждению данных.

## Шифрование

### Транспортный уровень
- Все соединения используют QUIC с TLS 1.3 (через Iroh).
- Дополнительное рукопожатие по протоколу Noise (паттерн XX) для аутентификации пиров.
- Результат: симметричное шифрование ChaCha20-Poly1305 с совершенной прямой секретностью (forward secrecy).
- Эфемерные ключи для каждой сессии — компрометация одной сессии не влияет на другие.

### Идентификация
- У каждого узла есть постоянная пара ключей Ed25519.
- Генерируется при первом запуске, хранится в системной связке ключей.
- ID узла = хеш публичного ключа.
- Нет центрального центра сертификации — модель сети доверия (web-of-trust).

### Что шифруется
| Данные | Зашифрованы | Примечания |
|---|---|---|
| Текст промпта | Да | Зашифрован при передаче; виден сиду (seed) в текущем эталонном потоке |
| Потоковый вывод текста | Да | Зашифрован при передаче; виден сиду, который его сгенерировал |
| Тензоры активации | Планируется | Актуально при разделенном инференсе (split inference) |
| Сообщения управления | Да | Все сообщения протокола внутри QUIC |
| Возможности пиров | Да | Обмен идет по зашифрованному каналу |

## Анализ угроз

### T1: Вредоносный узел-сид (текущий эталонный поток)
**Угроза**: Оператор сида читает промпт или ответ, так как воркер отправляет текст промпта сиду, и сид запускает всю модель.

**Текущий статус**: Это явная граница доверия, а не решенное свойство безопасности.

**Меры противодействия**:
- подключайте воркеры только к тем сидам, которым вы доверяете открытый текст промптов.
- сохраняйте транспорт зашифрованным, чтобы реле и пассивные наблюдатели не могли прочитать содержимое.
- переход к разделенному инференсу, чтобы промежуточные пиры не получали текст промпта.

### T2: Вредоносный узел пайплайна (целевой поток разделенного инференса)
**Угроза**: Узел в будущей цепочке пайплайна пытается извлечь промпт или ответ из промежуточных активаций.

**Целевая мера**: Узел на стадии пайплайна k должен видеть только тензор активации на выходе слоя k-1 и производить тензор активации для слоя k. Он не должен получать исходный текст промпта.

**Остаточный риск**: Тензоры активации могут допускать утечку информации о входных данных. Дифференциальная приватность, избыточность и аттестация являются предметом будущих работ.

### T3: Атака Сивиллы (Sybil Attack)
**Угроза**: Злоумышленник создает множество фиктивных узлов, чтобы доминировать в пайплайне.

**Меры противодействия**:
- Система репутации на основе наблюдаемого поведения (время в сети, корректность вычислений).
- Новые узлы начинают с низкой репутацией и ограниченными позициями в пайплайне.
- Критически важные слои (первый и последний) предпочтительно назначаются узлам с высокой репутацией.
- Ограничение скорости (rate limiting) присоединения новых узлов из одного диапазона IP.

### T4: Византийский инференс
**Угроза**: Вредоносный узел возвращает неверные тензоры активации.

**Меры противодействия (MVP)**: Принятие риска. В большинстве случаев явно неверный результат инференса заметен пользователю.

**Меры противодействия (будущее)**:
- Избыточные вычисления на критических слоях (2 узла вычисляют одни и те же слои, результаты сравниваются).
- Верифицируемые вычисления с использованием аттестации TEE (например, Secure Enclave в Apple Silicon).
- Статистическое обнаружение аномалий в распределениях тензоров активации.

### T5: Анализ трафика
**Угроза**: Наблюдатель мониторит паттерны зашифрованного трафика для получения информации об использовании.

**Меры противодействия**:
- QUIC мультиплексирует всю связь через одно соединение.
- Текущий трафик сид/воркер всё еще допускает утечку грубых метаданных о времени запросов и длине ответов.
- Заполнение (padding) сообщений управления до постоянного размера (опционально, не в MVP).

### T6: Компрометация реле-сервера
**Угроза**: Серверы-реле, используемые для бутстрапа, скомпрометированы.

**Последствия**: Минимальные. Реле-серверы только облегчают установку соединения. Они видят:
- Какие ID узлов подключаются (метаданные).
- Зашифрованные пакеты QUIC (не могут расшифровать).
- Они НЕ видят расшифрованные промпты или ответы.

**Меры противодействия**: Использование нескольких независимых операторов реле. Сеть продолжает работу без реле после заполнения DHT.

### T7: Отравление модели (Model Poisoning)
**Угроза**: Узел предоставляет модифицированную модель GGUF с внедренными вредоносными весами.

**Меры противодействия**:
- Файлы моделей проверяются по хешу SHA-256 на соответствие известным манифестам.
- Манифесты моделей распространяются через DHT с подписями издателей моделей.
- Узлы загружают модели только из проверенных источников (хеши HuggingFace).

### T8: Отказ в обслуживании (DoS)
**Угроза**: Узлы присоединяются, а затем перестают отвечать, нарушая работу инференса.

**Меры противодействия**:
- Тайм-аут сердцебиения (heartbeat) и ребалансировка — целевые свойства среды выполнения, а не полные гарантии текущей реализации.
- Локальный резервный режим (fallback) — цель проектирования для будущих клиентов с разделенным инференсом.
- Штраф к репутации для часто отключающихся узлов.
- Плавная деградация (graceful degradation) — основной принцип проектирования.
- Входящие запросы на инференс ограничены валидацией среды выполнения и фиксированным лимитом параллельного исполнения на сиде.
- Дублирующиеся значения `msg_id` протокола от одного пира отбрасываются в рамках окна защиты от повторов.

### T9: Раскрытие административного API
**Угроза**: Оператор привязывает локальный HTTP API к публичному интерфейсу без защиты, открывая доступ к `/status`, `/topology`, `/settlement` или `/chat`.

**Меры противодействия в текущей реализации**:
- По умолчанию демон привязывает HTTP API к `127.0.0.1`.
- Операторы всё же могут намеренно открыть его через `--bind 0.0.0.0`.
- Открытые административные маршруты могут быть защищены токеном доступа через `--api-token`.
- Размер JSON-тел запросов ограничен перед десериализацией для уменьшения злоупотреблений ресурсами на `/chat`.

**Остаточный риск**: Аутентификация по токену — это мера контроля оператора, а не взаимный TLS. Если токен утечет, API следует считать скомпрометированным до смены токена.

## Иерархия доверия

```
Наивысшее доверие: Ваше собственное устройство (телефон, ноутбук)
                   ↓
Доверенное:        Ваши собственные устройства в локальной сети (Mac Mini дома)
                   ↓
Полудоверенное:    WAN-пиры с высокой репутацией (месяцы работы)
                   ↓
Недоверенное:      Новые WAN-пиры (только что присоединились, без истории)
```

Назначение слоев должно следовать этой иерархии при наличии разделенного инференса:
- Первые и последние слои (наиболее чувствительные — видят входные эмбеддинги и выходные логиты) → ваши собственные устройства.
- Средние слои (видят только промежуточные активации) → могут быть назначены полудоверенным или недоверенным пирам.

## Гарантии приватности

**Что Forge гарантирует сегодня:**
- Промпты и ответы зашифрованы при передаче между напрямую подключенными пирами.
- Реле и пассивные сетевые наблюдатели не видят расшифрованное содержимое промптов или ответов.
- В пути передачи данных нет обязательного центрального сервера.
- Граница доверия сид/воркер является явной.

**Чего Forge не гарантирует сегодня:**
- Что сид не может прочитать промпт или ответ.
- Что разделенный инференс скрывает открытый текст от всех удаленных вычислительных провайдеров.
- Что неверный удаленный инференс обнаруживается автоматически.

**К каким гарантиям Forge стремится в будущем:**
- Узлы промежуточных стадий не получают открытый текст промптов.
- Тензоры активации зашифрованы при передаче между стадиями пайплайна.
- Видимость промптов сведена к минимальному набору доверенных граничных узлов.

Эти будущие гарантии зависят от реализации полноценного разделенного инференса. До тех пор Forge следует описывать как зашифрованный удаленный инференс с явной границей доверия.

## Экономические угрозы

### T10: Фальсификация CU

**Угроза**: Узел заявляет о CU, которые он не заработал, создавая фиктивные записи TradeRecord.

**Текущая мера**: Локальный леджер с защитой целостности HMAC-SHA256 предотвращает подделку файлов. Однако оператор узла всё равно может записывать произвольные сделки в свой собственный леджер.

**Целевая мера**: Протокол двойной подписи. Каждая запись TradeRecord должна быть подписана и провайдером, и потребителем. Узел не может начислить себе CU без подписи контрагента. Синхронизация через Gossip позволяет другим узлам независимо проверять обе подписи.

**Остаточный риск**: Сговор между провайдером и потребителем для создания фальшивых сделок. Это экономически иррационально — потребитель-соучастник ничего не выигрывает. Статистическое обнаружение аномалий в объеме и частоте сделок может выявлять подозрительные паттерны.

### T11: Злоупотребление бесплатным уровнем (Сивилла)

**Угроза**: Злоумышленник создает множество новых NodeId, чтобы многократно использовать бесплатный уровень в 1000 CU.

**Текущая мера**: Если в леджере существует более 100 неизвестных узлов (вклад = 0, потребление > 0), новые запросы по бесплатному уровню отклоняются. Каждая NodeId — это пара ключей Ed25519; создать их легко, но они отслеживаемы.

**Целевая мера**: Proof of Work при регистрации узла (небольшие вычислительные затраты на создание новой личности) или вход на основе стейка (новые узлы должны внести вклад вычислениями, прежде чем начать потреблять).

### T12: Расхождение леджеров (Divergence)

**Угроза**: Разные узлы имеют несовместимые представления об одних и тех же сделках, что ведет к экономической несогласованности.

**Текущая мера**: Каждый узел поддерживает свое локальное представление. Гарантии согласованности между узлами нет.

**Целевая мера**: Синхронизируемые через gossip записи TradeRecord с двойной подписью. Обе стороны создают идентичные подписанные записи. Любой узел, получающий обновление через gossip, может проверить подписи и отклонить несоответствия. Периодическая фиксация сводок в Bitcoin (OP_RETURN) обеспечивает опциональный неизменяемый аудиторский след.

### T13: Манипулирование рынком

**Угроза**: Узел искусственно завышает факторы спроса или предложения для манипулирования ценами.

**Текущая мера**: Рыночная цена рассчитывается локально на основе собственных наблюдений каждого узла. Ни один узел не может заставить другой узел принять его цену.

**Целевая мера**: Сигналы о ценах на основе gossip, взвешенные по репутации. Наблюдения узлов с высокой репутацией имеют больший вес. Новые или низкорейтинговые узлы не могут существенно влиять на сетевое ценообразование.

### T14: Атака на качество инференса

**Угроза**: Провайдер возвращает низкокачественный или усеченный инференс, чтобы заработать CU, не выполняя полных вычислений.

**Текущая мера**: Принятие риска. В большинстве случаев явно неверные выходные данные заметны потребителю.

**Целевая мера**: Верификация качества на стороне потребителя. Потребитель может повторно прогнать небольшую выборку токенов локально, чтобы убедиться, что вывод провайдера консистентен. Штраф к репутации для провайдеров, чьи результаты не проходят выборочную проверку.
