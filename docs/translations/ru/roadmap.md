# Forge — План развития (Roadmap)

## Фаза 1: Локальный инференс ✅

- `forge-core`: Система типов (NodeId, LayerRange, ModelManifest, PeerCapability)
- `forge-infer`: движок llama.cpp, загрузчик GGUF, потоковая генерация токенов
- `forge-node`: HTTP API (/chat, /chat/stream, /health)
- `forge-cli`: команда `forge chat` с автоматической загрузкой моделей

## Фаза 2: Протокол P2P ✅

- `forge-net`: транспорт Iroh, шифрование Noise, соединения с пирами
- `forge-proto`: 14 типов сообщений протокола (bincode + префикс длины)
- `forge-node`: пайплайн Seed/Worker, запрос/ответ инференса
- Интеграционные тесты: 2 узла обмениваются Hello + несколькими сообщениями

## Фаза 3: Удаленный инференс + Леджер оператора ✅

- `forge-ledger`: учет CU, выполнение сделок, репутация, доход, рыночные цены
- `forge-node`: леджер, интегрированный в пайплайн инференса
- Проверка баланса CU перед началом инференса
- Запись сделок после завершения
- Целостность леджера на базе HMAC-SHA256

## Фаза 4: Экономический API ✅

- OpenAI-совместимый API: `POST /v1/chat/completions`, `GET /v1/models`
- Учет CU: каждый инференс записывает сделку с расширением `x_forge`
- Эндпоинты бюджета агента: `GET /v1/forge/balance`, `GET /v1/forge/pricing`
- Мост расчетов CU→Lightning: `forge settle --pay`
- Автоматическое разрешение моделей сида через HF Hub
- Корректное завершение работы (Ctrl-C) с сохранением леджера

## Фаза 5: Интеграция форка mesh-llm (следующая)

**Цель:** Заменить слой инференса Forge на проверенный распределенный движок mesh-llm.

| Результат | Описание |
|---|---|
| Форк mesh-llm | Создание forge как форка mesh-llm с экономическим слоем |
| Интеграция forge-ledger | Подключение записи CU к пайплайну инференса mesh-llm |
| Сохранение эконом. API | Поддержка эндпоинтов /v1/forge/* в новом коде |
| Расширение веб-консоли | Добавление видимости баланса CU и сделок в консоль mesh-llm |
| Пайплайн + MoE | Наследование параллелизма пайплайна и шардинга экспертов mesh-llm |
| Поиск через Nostr | Наследование механизма поиска публичных сетей mesh-llm |
| CREDITS.md | Документирование авторства mesh-llm |

## Фаза 6: Доказательство полезной работы (Proof of Useful Work)

**Цель:** Сделать претензии на CU верифицируемыми во всей сети.

| Результат | Описание |
|---|---|
| Протокол двойной подписи | И провайдер, и потребитель подписывают каждую запись TradeRecord |
| Синхронизация Gossip | Распространение подписанных сделок по всей сети |
| Обнаружение мошенничества | Отклонение неподписанных или несовпадающих сделок |
| Gossip репутации | Обмен показателями репутации между пирами |
| Устойчивость к сговору | Статистическое обнаружение аномалий в паттернах сделок |

## Фаза 7: Внешние мосты

**Цель:** Позволить операторам конвертировать CU во внешнюю стоимость.

| Результат | Описание |
|---|---|
| Мост Lightning | Автоматизированные расчеты CU→sats через LDK |
| Адаптер стейблкоинов | Конвертация CU→USDC/USDT |
| Интерфейс фиатного адаптера | Спецификация для расчетов банковским переводом |
| Сервис обменных курсов | Публичные фиды курсов CU/BTC и CU/USD |
| Фиксация в Bitcoin | Опционально: периодический корень Меркла → OP_RETURN для неизменяемого аудита |

## Фаза 8: Автономная экономика агентов

**Цель:** Позволить ШИ-агентам самостоятельно управлять своим жизненным циклом вычислений.

| Результат | Описание |
|---|---|
| Бюджетные политики | Устанавливаемые человеком лимиты трат на одного агента |
| Автономная торговля | Агент сам решает, когда покупать/продавать вычисления |
| Мультимодельная маршрутизация | Агент выбирает модель на основе баланса стоимость/качество |
| Самосиление | Агент зарабатывает CU → покупает доступ к большей модели → зарабатывает еще больше CU |
| Межагентская экономика | Агенты торгуют специализированными вычислениями (код-модель против чат-модели) |

## Долгосрочные цели

| Веха | Описание |
|---|---|
| Релиз SDK | forge-node как встраиваемая библиотека Rust со стабильным API |
| Протокол v2 | Уроки v1, обратно совместимая эволюция |
| Кросс-архитектурность | Поддержка NVIDIA GPU, AMD ROCm, RISC-V (через mesh-llm) |
| Федеративное обучение | Распределенная донастройка (fine-tuning), а не только инференс |
| Деривативы на вычисления | Форвардные контракты на будущие вычислительные мощности |

> Протокол — это платформа. Вычисления — это валюта.
