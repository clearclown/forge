# Forge — Архитектура

## Обзор

Forge — это двухслойная система: **инференс** и **экономика**.

Слой инференса отвечает за распределение моделей, ячеистую сеть (mesh) и обслуживание API. Он построен на [mesh-llm](https://github.com/michaelneale/mesh-llm).

Экономический слой отвечает за учет CU, запись сделок, ценообразование и бюджеты агентов. Это оригинальный вклад Forge.


## Слой инференса (mesh-llm)

Слой инференса отвечает за:

- **Ячеистая сеть (Mesh)**: QUIC-соединения на базе iroh с шифрованием Noise
- **Поиск пиров**: Nostr-реле для публичных сетей, mDNS для локальных сетей
- **Распределение моделей**: Параллелизм пайплайна для плотных моделей, шардинг экспертов для MoE
- **Выполнение инференса**: llama.cpp через подпроцессы llama-server и rpc-server
- **Обслуживание API**: OpenAI-совместимые эндпоинты `/v1/chat/completions` и `/v1/models`

Forge наследует всё это от mesh-llm. Слой инференса ничего не знает о CU, сделках или ценообразовании.

## Экономический слой (Forge)

Экономический слой находится над инференсом и отвечает за:

### forge-ledger — Экономический движок

```rust
pub struct ComputeLedger {
    balances: HashMap<NodeId, NodeBalance>,
    work_log: Vec<WorkUnit>,
    trade_log: Vec<TradeRecord>,
    price: MarketPrice,
}
```

Основные обязанности:
- Отслеживание баланса CU каждого узла (вклад, потребление, резерв)
- Запись каждой сделки по инференсу (провайдер, потребитель, сумма CU, токены)
- Расчет динамических рыночных цен на основе спроса и предложения
- Начисление дохода узлам-участникам
- Экспорт отчетов о расчетах для внепротокольных мостов
- Сохранение снимков состояния на диск с проверкой целостности HMAC-SHA256

### forge-verify — Доказательство полезной работы (в разработке)

Гарантирует легитимность претензий на CU:
- Протокол двойной подписи: и провайдер, и потребитель подписывают каждую запись TradeRecord
- Синхронизация через Gossip: подписанные сделки распространяются по сети
- Верификация: любой узел может проверить обе подписи
- Обнаружение мошенничества: несовпадающие или неподписанные сделки отклоняются

### forge-bridge — Внешние расчеты (опционально)

Конвертирует CU во внешнюю стоимость для операторов:
- Bitcoin Lightning: CU → msats через настраиваемый обменный курс
- Stablecoin: CU → USDC/USDT через адаптер
- Фиат: CU → банковский перевод через дашборд оператора

Слой моста находится вне основного протокола. Разные операторы могут использовать разные мосты.

### Поверхность API

| Маршрут | Слой | Описание |
|-------|-------|-------------|
| `POST /v1/chat/completions` | Инференс + Экономика | Запуск инференса, запись сделки CU |
| `GET /v1/models` | Инференс | Список загруженных моделей |
| `GET /v1/forge/balance` | Экономика | Баланс CU, репутация |
| `GET /v1/forge/pricing` | Экономика | Рыночная цена, оценка стоимости |
| `GET /status` | Экономика | Рыночная цена, статистика сети, недавние сделки |
| `GET /topology` | Инференс | Манифест модели, пиры, план шардинга |
| `GET /settlement` | Экономика | Экспортируемая история сделок |
| `GET /health` | Инференс | Базовая проверка состояния |

## Поток данных

### Инференс с учетом CU

```
Потребитель отправляет запрос
    ↓
API получает POST /v1/chat/completions
    ↓
Леджер проверяет: can_afford(consumer, estimated_cost)?
    ↓ да
Слой инференса выполняется (llama-server / rpc-server)
    ↓
Токены стримятся обратно потребителю
    ↓
Леджер записывает сделку:
  - provider.contributed += cu_cost
  - consumer.consumed += cu_cost
  - trade_log.push(TradeRecord)
    ↓
Ответ содержит x_forge: { cu_cost, effective_balance }
```

### Экспорт расчетов

```
Оператор запускает: forge settle --hours 24
    ↓
API читает trade_log за временное окно
    ↓
Агрегирует по узлам: gross_earned, gross_spent, net_cu
    ↓
Экспортирует JSON-отчет с опциональной справочной ценой
    ↓
Оператор использует адаптер моста для конвертации чистых CU в BTC/фиат
```

## Модель безопасности

```
Уровень 0: Основная сеть Bitcoin ← Опциональная фиксация (в будущем)
Уровень 1: Двойные подписи     ← Провайдер + потребитель подписывают сделку
Уровень 2: Леджер HMAC-SHA256  ← Локальная защита целостности
Уровень 3: iroh (QUIC + Noise) ← Шифрование транспорта
Уровень 4: Выполнение инференса ← Модель работает локально у провайдера
```

Каждый уровень защищает от разных угроз:
- Уровень 4: Целостность модели (проверка хеша GGUF)
- Уровень 3: Конфиденциальность транспорта (прослушивание)
- Уровень 2: Локальное вмешательство (изменение файлов)
- Уровень 1: Сетевое мошенничество (ложные претензии на CU)
- Уровень 0: Историческая неизменяемость (опциональный якорь в Bitcoin)

## Зависимости пакетов (crates)

```
forge-core ← общие типы (NodeId, CU, Config)
    ↑
forge-ledger ← экономический движок (сделки, цены, доход)
    ↑
forge-lightning ← внешний мост (кошелек LDK, CU↔sats)
    ↑
forge-node ← оркестратор (HTTP API, пайплайн, интеграция леджера)
    ↑
forge-cli ← эталонный CLI (chat, seed, worker, settle)

forge-net ← P2P транспорт (iroh, QUIC, Noise, mDNS)
forge-proto ← протокольные сообщения (bincode, 14 типов полезной нагрузки)
forge-infer ← движок инференса (llama.cpp, загрузчик GGUF)
forge-shard ← планировщик топологии (назначение слоев, ребалансировка)
```
