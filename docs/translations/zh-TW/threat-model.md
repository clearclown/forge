# Forge — 威脅模型

## 安全目標

1. **傳輸中的機密性**: 被動觀察者和中繼節點不應讀取提示詞或回應
2. **經過身份驗證的對等節點**: 每個直接連接都應綁定到一個加密的節點身份
3. **有限信任**: 系統應明確哪些節點看到明文，哪些節點僅看到中間狀態
4. **可用性**: 單個節點故障應使服務降級，而不是默默地損壞數據

## 加密

### 傳輸層
- 所有連接使用帶 TLS 1.3 的 QUIC (透過 Iroh)
- 額外的 Noise 協定握手 (XX 模式) 用於節點身份驗證
- 結果: 具有前向安全性的 ChaCha20-Poly1305 對稱加密
- 使用每會話臨時金鑰 — 一個會話的洩露不會影響其他會話

### 身份 (Identity)
- 每個節點都有一個持久的 Ed25519 金鑰對
- 在首次啟動時生成，存儲在平台金鑰鏈中
- 節點 ID = 公鑰哈希
- 無中央證書頒發機構 — 信任網絡 (web-of-trust) 模型

### 加密內容
| 數據 | 已加密 | 備註 |
|---|---|---|
| 提示詞文本 | 是 | 傳輸中加密；在當前參考流程中對播種節點可見 |
| 流式文本輸出 | 是 | 傳輸中加密；對生成它的播種節點可見 |
| 激活張量 | 計劃中 | 一旦運行時激活拆分推理，此項即相關 |
| 控制消息 | 是 | QUIC 內的所有協定消息 |
| 節點能力 | 是 | 透過加密通道交換 |

## 威脅分析

### T1: 惡意播種節點 (當前參考流程)
**威脅**: 播種節點操作員讀取提示詞或回應，因為工人將提示詞文本發送給播種節點，且播種節點運行完整模型。

**現狀**: 這是一個顯式的信任邊界，而非已解決的安全屬性。

**緩解措施**:
- 僅將工人連接到您信任能夠處理明文提示詞的播種節點
- 保持傳輸加密，使中繼和被動觀察者無法讀取內容
- 向拆分推理遷移，使中間階段節點不接收明文提示詞

### T2: 惡意流水線節點 (目標拆分推理流程)
**威脅**: 未來流水線中的某個節點試圖從中間激活中提取提示詞或回應。

**緩解目標**: 流水線第 k 階段的節點應僅看到第 k-1 層的激活張量輸出，並生成第 k 層的激活張量。它不應收到原始提示詞文本。

**殘留風險**: 激活張量可能會洩露有關輸入的資訊。差分隱私、冗餘和認證 (attestation) 仍是未來工作。

### T3: 女巫攻擊 (Sybil Attack)
**威脅**: 攻擊者創建許多虛假節點以控制流水線。

**緩解措施**:
- 基於觀察到的行為（在線時間、計算正確性）的聲譽系統
- 新節點從低聲譽開始，且流水線位置受限
- 關鍵層（第一層和最後一層）優先分配給高聲譽節點
- 對來自同一 IP 範圍的新節點加入進行速率限制

### T4: 拜占庭推理
**威脅**: 惡意節點返回錯誤的激活張量。

**緩解措施 (MVP)**: 接受風險。對於大多數用例，用戶可以檢測到細微錯誤的推理結果。

**緩解措施 (未來)**:
- 在關鍵層進行冗余計算（2 個節點計算相同層級並對比）
- 使用 TEE 認證 (Apple Silicon Secure Enclave) 進行可驗證計算
- 對激活張量分佈進行統計異常檢測

### T5: 流量分析
**威脅**: 觀察者監控加密通訊模式以推斷使用情況。

**緩解措施**:
- QUIC 在單個連接上複用所有通訊
- 當前的播種/工人流量仍會洩露粗略的請求時間和回應長度元數據
- 控制消息填充至固定大小（可選，不在 MVP 中）

### T6: 中繼伺服器被攻陷
**威脅**: 引導啟動中繼伺服器被攻陷。

**影響**: 極小。中繼伺服器僅協助建立連接。它們能看到：
- 哪些節點 ID 正在連接（元數據）
- 加密的 QUIC 數據包（無法解密）
- 它們**看不到**解密後的提示詞或回應

**緩解措施**: 多個獨立的中繼操作員。一旦 DHT 填充，網絡無需中繼即可繼續運行。

### T7: 模型中毒
**威脅**: 某個節點提供帶有後門權重的修改版 GGUF 模型。

**緩解措施**:
- 模型文件透過 SHA-256 哈希針對已知良好的清單進行驗證
- 模型清單透過 DHT 發布，並帶有模型發布者的簽名
- 節點僅從驗證過的來源（HuggingFace 哈希）加載模型

### T8: 拒絕服務 (DoS)
**威脅**: 節點加入後變得無響應，干擾推理。

**緩解措施**:
- 心跳超時和重新平衡是目標運行時屬性，而非當前實現的完全保證
- 將本地回退作為未來拆分推理客戶端的设计目標
- 對頻繁斷開連接的節點進行聲譽懲罰
- 優雅降級是核心設計原則
- 入站推理請求受運行時驗證和播種節點上固定的併發執行限制約束
- 來自同一節點的重複協定 `msg_id` 值在有限重放窗口內被丟棄

### T9: 管理 API 暴露
**威脅**: 操作員將本地 HTTP API 綁定到公共接口且無保護，暴露了 `/status`、`/topology`、`/settlement` 或 `/chat`。

**當前實現中的緩解措施**:
- 守護進程預設將 HTTP API 綁定到 `127.0.0.1`
- 操作員仍可以使用 `--bind 0.0.0.0` 有意公開它
- 公開的管理路由可以透過 `--api-token` 使用持有者令牌 (bearer token) 保護
- JSON 請求體在反序列化前受大小限制，以減少 `/chat` 上的分配濫用

**殘留風險**: 持有者令牌驗證是操作員控制，而非雙向 TLS。如果令牌洩露，在輪換之前應將 API 視為已被攻陷。

## 信任層級

```
最信任:          您自己的設備 (手機, 筆記本)
                 ↓
信任:            區域網內您自己的設備 (家中的 Mac Mini)
                 ↓
半信任:          高聲譽的廣域網節點 (數月的在線時間)
                 ↓
不信任:          新的廣域網節點 (剛加入, 無歷史記錄)
```

一旦拆分推理存在，層級分配應遵循此層級結構：
- 第一層和最後一層（最敏感 — 看到輸入嵌入和輸出 logits）→ 您自己的設備
- 中間層（僅看到中間激活）→ 可分配給半信任或不信任節點

## 隱私保證

**Forge 今天的保證:**
- 提示詞和回應在直接連接的節點之間傳輸加密
- 中繼和被動網絡觀察者看不到解密後的提示詞或回應內容
- 數據路徑中沒有強制性的中央伺服器
- 當前的播種/工人信任邊界是顯式的

**Forge 今天不保證的內容:**
- 播種節點無法讀取提示詞或回應
- 拆分推理對所有遠端計算提供者隱藏明文
- 自動檢測到不正確的遠端推理

**Forge 以後旨在保證的內容:**
- 中間階段節點不接收明文提示詞
- 激活張量在流水線階段之間傳輸加密
- 提示詞可見性降低到最小的一組信任邊界節點

這些後續保證取決於實際發布拆分推理。在此之前，Forge 應被描述為具有誠實信任邊界的加密遠端推理。

## 經濟威脅

### T10: CU 偽造

**威脅**: 某個節點透過編造 TradeRecords 聲明其未賺取的 CU。

**當前緩解措施**: 具有 HMAC-SHA256 完整性保護的本地帳本可防止文件級篡改。然而，節點操作員仍可以向其自己的帳本中寫入任意交易。

**目標緩解措施**: 雙重簽名協定。每個 TradeRecord 必須由提供者和消費者共同簽名。節點在沒有交易對手簽名的情況下無法給自己信用 CU。Gossip 同步意味著其他節點可以獨立驗證雙方簽名。

**殘留風險**: 提供者和消費者之間串通創建虛假交易。這在經濟上是不合理的——串通的消費者得不到任何好處。對交易量和頻率的統計異常檢測可以標記可疑模式。

### T11: 免費層級濫用 (女巫攻擊)

**威脅**: 攻擊者創建許多新的 NodeId 以重複利用 1,000 CU 的免費層級。

**當前緩解措施**: 如果帳本中存在 100 個以上的未知節點（貢獻 = 0, 消耗 > 0），新的免費層級請求將被拒絕。每個 NodeId 都是一個 Ed25519 金鑰對 — 創建成本低但可追蹤。

**目標緩解措施**: 節點註冊的工作量證明（創建新身份的小額計算成本），或基於質押的准入（新節點在消費前必須貢獻計算資源）。

### T12: 帳本分歧

**威脅**: 不同節點對同一筆交易有不相容的看法，導致經濟不一致。

**當前緩解措施**: 各個節點維護自己的本地視圖。節點之間沒有一致性保證。

**目標緩解措施**: Gossip 同步的雙重簽名 TradeRecords。雙方產生相同的簽名記錄。任何收到 gossip 更新的節點都可以驗證簽名並拒絕不一致。定期向比特幣錨定摘要 (OP_RETURN) 可提供可選的不可篡改審計追蹤。

### T13: 市場操縱

**威脅**: 某個節點人為地誇大需求或供應因子以操縱定價。

**當前緩解措施**: 市場價格根據每個節點自己的觀察在本地計算。任何單個節點都無法強迫另一個節點採用其價格。

**目標緩解措施**: 基於 gossip 的聲譽加權價格信號。高聲譽節點的觀察具有更大權重。新節點或低聲譽節點無法顯著影響全網定價。

### T14: 推理品質攻擊

**威脅**: 提供者返回低品質或被截斷的推理，以在未完成全部計算的情況下賺取 CU。

**當前緩解措施**: 接受風險。對於大多數用例，消費者可以檢測到明顯錯誤的輸出。

**目標緩解措施**: 消費者端品質驗證。消費者可以本地重新運行一小部分 token 示例，以驗證提供者的輸出是否一致。對輸出未能通過抽查的提供者進行聲譽處罰。
