# Forge — 威胁模型

## 安全目标

1. **传输中的机密性**: 被动观察者和中继节点不应读取提示词或响应
2. **经过身份验证的对等节点**: 每个直接连接都应绑定到一个加密的节点身份
3. **有限信任**: 系统应明确哪些节点看到明文，哪些节点仅看到中间状态
4. **可用性**: 单个节点故障应使服务降级，而不是默默地损坏数据

## 加密

### 传输层
- 所有连接使用带 TLS 1.3 的 QUIC (通过 Iroh)
- 额外的 Noise 协议握手 (XX 模式) 用于节点身份验证
- 结果: 具有前向安全性的 ChaCha20-Poly1305 对称加密
- 使用每会话临时密钥 — 一个会话的泄露不会影响其他会话

### 身份 (Identity)
- 每个节点都有一个持久的 Ed25519 密钥对
- 在首次启动时生成，存储在平台密钥链中
- 节点 ID = 公钥哈希
- 无中央证书颁发机构 — 信任网络 (web-of-trust) 模型

### 加密内容
| 数据 | 已加密 | 备注 |
|---|---|---|
| 提示词文本 | 是 | 传输中加密；在当前参考流程中对播种节点可见 |
| 流式文本输出 | 是 | 传输中加密；对生成它的播种节点可见 |
| 激活张量 | 计划中 | 一旦运行时激活拆分推理，此项即相关 |
| 控制消息 | 是 | QUIC 内的所有协议消息 |
| 节点能力 | 是 | 通过加密通道交换 |

## 威胁分析

### T1: 恶意播种节点 (当前参考流程)
**威胁**: 播种节点操作员读取提示词或响应，因为工人将提示词文本发送给播种节点，且播种节点运行完整模型。

**现状**: 这是一个显式的信任边界，而非已解决的安全属性。

**缓解措施**:
- 仅将工人连接到您信任能够处理明文提示词的播种节点
- 保持传输加密，使中继和被动观察者无法读取内容
- 向拆分推理迁移，使中间阶段节点不接收明文提示词

### T2: 恶意流水线节点 (目标拆分推理流程)
**威胁**: 未来流水线中的某个节点试图从中间激活中提取提示词或响应。

**缓解目标**: 流水线第 k 阶段的节点应仅看到第 k-1 层的激活张量输出，并生成第 k 层的激活张量。它不应收到原始提示词文本。

**残留风险**: 激活张量可能会泄露有关输入的信息。差分隐私、冗余和认证 (attestation) 仍是未来工作。

### T3: 女巫攻击 (Sybil Attack)
**威胁**: 攻击者创建许多虚假节点以控制流水线。

**缓解措施**:
- 基于观察到的行为（在线时间、计算正确性）的声誉系统
- 新节点从低声誉开始，且流水线位置受限
- 关键层（第一层和最后一层）优先分配给高声誉节点
- 对来自同一 IP 范围的新节点加入进行速率限制

### T4: 拜占庭推理
**威胁**: 恶意节点返回错误的激活张量。

**缓解措施 (MVP)**: 接受风险。对于大多数用例，用户可以检测到细微错误的推理结果。

**缓解措施 (未来)**:
- 在关键层进行冗余计算（2 个节点计算相同层级并对比）
- 使用 TEE 认证 (Apple Silicon Secure Enclave) 进行可验证计算
- 对激活张量分布进行统计异常检测

### T5: 流量分析
**威胁**: 观察者监控加密通信模式以推断使用情况。

**缓解措施**:
- QUIC 在单个连接上复用所有通信
- 当前的播种/工人流量仍会泄露粗略的请求时间和响应长度元数据
- 控制消息填充至固定大小（可选，不在 MVP 中）

### T6: 中继服务器被攻陷
**威胁**: 引导启动中继服务器被攻陷。

**影响**: 极小。中继服务器仅协助建立连接。它们能看到：
- 哪些节点 ID 正在连接（元数据）
- 加密的 QUIC 数据包（无法解密）
- 它们**看不到**解密后的提示词或响应

**缓解措施**: 多个独立的中继操作员。一旦 DHT 填充，网络无需中继即可继续运行。

### T7: 模型中毒
**威胁**: 某个节点提供带有后门权重的修改版 GGUF 模型。

**缓解措施**:
- 模型文件通过 SHA-256 哈希针对已知良好的清单进行验证
- 模型清单通过 DHT 发布，并带有模型发布者的签名
- 节点仅从验证过的来源（HuggingFace 哈希）加载模型

### T8: 拒绝服务 (DoS)
**威胁**: 节点加入后变得无响应，干扰推理。

**缓解措施**:
- 心跳超时和重新平衡是目标运行时属性，而非当前实现的完全保证
- 将本地回退作为未来拆分推理客户端的设计目标
- 对频繁断开连接的节点进行声誉惩罚
- 优雅降级是核心设计原则
- 入站推理请求受运行时验证和播种节点上固定的并发执行限制约束
- 来自同一节点的重复协议 `msg_id` 值在有限重放窗口内被丢弃

### T9: 管理 API 暴露
**威胁**: 操作员将本地 HTTP API 绑定到公共接口且无保护，暴露了 `/status`、`/topology`、`/settlement` 或 `/chat`。

**当前实现中的缓解措施**:
- 守护进程默认将 HTTP API 绑定到 `127.0.0.1`
- 操作员仍可以使用 `--bind 0.0.0.0` 有意公开它
- 公开的管理路由可以通过 `--api-token` 使用持有者令牌 (bearer token) 保护
- JSON 请求体在反序列化前受大小限制，以减少 `/chat` 上的分配滥用

**残留风险**: 持有者令牌验证是操作员控制，而非双向 TLS。如果令牌泄露，在轮换之前应将 API 视为已被攻陷。

## 信任层级

```
最信任:          您自己的设备 (手机, 笔记本)
                 ↓
信任:            局域网内您自己的设备 (家中的 Mac Mini)
                 ↓
半信任:          高声誉的广域网节点 (数月的在线时间)
                 ↓
不信任:          新的广域网节点 (刚加入, 无历史记录)
```

一旦拆分推理存在，层级分配应遵循此层级结构：
- 第一层和最后一层（最敏感 — 看到输入嵌入和输出 logits）→ 您自己的设备
- 中间层（仅看到中间激活）→ 可分配给半信任或不信任节点

## 隐私保证

**Forge 今天的保证:**
- 提示词和响应在直接连接的节点之间传输加密
- 中继和被动网络观察者看不到解密后的提示词或响应内容
- 数据路径中没有强制性的中央服务器
- 当前的播种/工人信任边界是显式的

**Forge 今天不保证的内容:**
- 播种节点无法读取提示词或响应
- 拆分推理对所有远程计算提供者隐藏明文
- 自动检测到不正确的远程推理

**Forge 以后旨在保证的内容:**
- 中间阶段节点不接收明文提示词
- 激活张量在流水线阶段之间传输加密
- 提示词可见性降低到最小的一组信任边界节点

这些后续保证取决于实际发布拆分推理。在此之前，Forge 应被描述为具有诚实信任边界的加密远程推理。

## 经济威胁

### T10: CU 伪造

**威胁**: 某个节点通过编造 TradeRecords 声明其未赚取的 CU。

**当前缓解措施**: 具有 HMAC-SHA256 完整性保护的本地账本可防止文件级篡改。然而，节点操作员仍可以向其自己的账本中写入任意交易。

**目标缓解措施**: 双重签名协议。每个 TradeRecord 必须由提供者和消费者共同签名。节点在没有交易对手签名的情况下无法给自己信用 CU。Gossip 同步意味着其他节点可以独立验证双方签名。

**残留风险**: 提供者和消费者之间串通创建虚假交易。这在经济上是不合理的——串通的消费者得不到任何好处。对交易量和频率的统计异常检测可以标记可疑模式。

### T11: 免费层级滥用 (女巫攻击)

**威胁**: 攻击者创建许多新的 NodeId 以重复利用 1,000 CU 的免费层级。

**当前缓解措施**: 如果账本中存在 100 个以上的未知节点（贡献 = 0, 消耗 > 0），新的免费层级请求将被拒绝。每个 NodeId 都是一个 Ed25519 密钥对 — 创建成本低但可追踪。

**目标缓解措施**: 节点注册的工作量证明（创建新身份的小额计算成本），或基于质押的准入（新节点在消费前必须贡献计算资源）。

### T12: 账本分歧

**威胁**: 不同节点对同一笔交易有不兼容的看法，导致经济不一致。

**当前缓解措施**: 每个节点维护自己的本地视图。节点之间没有一致性保证。

**目标缓解措施**: Gossip 同步的双重签名 TradeRecords。双方产生相同的签名记录。任何收到 gossip 更新的节点都可以验证签名并拒绝不一致。定期向比特币锚定摘要 (OP_RETURN) 可提供可选的不可篡改审计追踪。

### T13: 市场操纵

**威胁**: 某个节点人为地夸大需求或供应因子以操纵定价。

**当前缓解措施**: 市场价格根据每个节点自己的观察在本地计算。任何单个节点都无法强迫另一个节点采用其价格。

**目标缓解措施**: 基于 gossip 的声誉加权价格信号。高声誉节点的观察具有更大权重。新节点或低声誉节点无法显著影响全网定价。

### T14: 推理质量攻击

**威胁**: 提供者返回低质量或被截断的推理，以在未完成全部计算的情况下赚取 CU。

**当前缓解措施**: 接受风险。对于大多数用例，消费者可以检测到明显错误的输出。

**目标缓解措施**: 消费者端质量验证。消费者可以本地重新运行一小部分 token 示例，以验证提供者的输出是否一致。对输出未能通过抽查的提供者进行声誉处罚。
