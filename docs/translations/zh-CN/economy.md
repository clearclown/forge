# Forge — 经济模型

## 计算本位制 (Compute Standard)

每种货币体系都以稀缺性为支撑：

| 时代 | 标准 | 支撑 |
|-----|----------|---------|
| 古代 | 金/银 | 地质稀缺性 |
| 1944–1971 | 布雷顿森林体系 | 美元挂钩黄金 |
| 1971–至今 | 石油美元 | 石油需求 + 军事力量 |
| 2009–至今 | 比特币 | 燃烧在 SHA-256 上的电力 |
| **Forge** | **计算本位制** | **花费在有用推理上的电力** |

Forge 引入了计算本位制：价值单位由执行有用计算的真实能源支出支撑。与比特币的工作量证明不同，Forge 中花费的每一焦耳都会产生真实的智能。

## CU: 原生货币

### 什么是 CU

**1 CU = 验证过的 10 亿次浮点运算 (FLOPs) 的推理工作量。**

CU 不是加密货币。它不是区块链上的代币。它是一个代表已执行的真实计算量的核算单位。CU 具有价值，是因为它是对未来计算能力的请求权——如果你通过提供推理赚取了 CU，你可以花费它来换取推理服务。

### 为什么选择 CU，而不是比特币

| 属性 | CU | 比特币 |
|----------|-----|---------|
| **价值支撑** | 有用计算 (内在价值) | 哈希计算 (人为价值) |
| **结算速度** | 即时 (本地账本) | 秒到分钟 (Lightning/链上) |
| **交易成本** | 零 | 通道费、链上费 |
| **外部依赖** | 无 | 比特币网络健康状况 |
| **量子风险** | 无 (无加密谜题) | SHA-256 / ECDSA 易受攻击 |
| **收益产生** | 是 (空闲硬件赚取 CU) | 否 (钱包中的 BTC 不产生收益) |

CU 是**主要**结算单位。比特币、稳定币和法币是通过协议外的桥接适配器提供的可选**离场通道**。

### 作为生产性资产的 CU

```
公寓楼                      Forge 上的 Mac Mini
───────────────────         ──────────────────
资产: 建筑物                 资产: 计算硬件
成本: 维护                   成本: 电力
收入: 租金                   收入: 推理产生的 CU
收益: 租金 - 维护            收益: 赚取的 CU - 电力成本
空闲 = 收入损失              空闲 = 潜力浪费
```

Forge 上的计算设备不同于钱包里的比特币（静态价值，无收益）。它就像出租物业——通过执行有用工作产生收入。

## 交易模型

### 交易执行

每次推理都会在两方之间产生一笔交易：

```rust
pub struct TradeRecord {
    pub provider: NodeId,       // 谁运行了推理
    pub consumer: NodeId,       // 谁请求了推理
    pub cu_amount: u64,         // 转移的 CU 金额
    pub tokens_processed: u64,  // 完成的工作量
    pub timestamp: u64,
    pub model_id: String,
}
```

交易由双方记录。在当前实现中，每个节点维护一个本地账本。目标实现将增加双重签名和 gossip 同步。

### 动态定价

CU 价格根据本地供需关系浮动：

```
有效价格 = 基础 CU/token × 需求因子 / 供应因子
```

- **空闲节点越多** → 供应因子上升 → 价格下跌
- **推理请求越多** → 需求因子上升 → 价格上涨
- 每个节点观察自己的市场状况。没有全球订单簿。

### 免费层级

没有贡献历史的新节点将获得 1,000 CU。这让任何人都能立即使用网络。免费层级从第一次请求开始消耗，且不会重置。

女巫攻击 (Sybil) 缓解：如果账本中存在超过 100 个从未贡献过的未知节点（贡献 = 0, 消耗 > 0），新的免费层级请求将被拒绝。

## 有用工作证明 (Proof of Useful Work)

### 概念

比特币的工作量证明：“我燃烧电力计算了 SHA-256 哈希。这是证明它的随机数 (nonce)。”

Forge 的有用工作证明：“我燃烧电力运行了 LLM 推理。这是响应，这是消费者确认收到响应的签名。”

关键区别：比特币的证明是自生成的（任何矿工都可以产生有效的哈希）。Forge 的证明需要一个**交易对手**——真正需要该推理的人。你无法伪造需求。

### 验证协议 (目标)

```
1. 消费者向提供者发送 InferenceRequest
2. 提供者执行推理，将 token 流回
3. 消费者收到 token，计算响应哈希
4. 双方为 TradeRecord 签名：
   - 提供者签名：“我计算了这些”
   - 消费者签名：“我收到了这些”
5. 双重签名的 TradeRecord 通过 gossip 同步到网络
6. 任何节点都可以验证双方签名
```

节点无法在没有协作对手的情况下增加其 CU 余额。共谋是可能的，但在经济上是不合理的——共谋的消费者通过签署虚假交易得不到任何好处。

### 当前实现

当前的参考实现使用具有 HMAC-SHA256 完整性保护的本地账本。双重签名和 gossip 是下一步。

## 收益与声誉

### 收益

保持在线并贡献计算资源的节点将赚取收益：

```
收益 CU = 贡献 CU × 0.001 × 声誉 × 在线时长 (小时)
```

在声誉为 1.0 时，一个贡献了 10,000 CU 的节点在 8 小时的夜间可以赚取 80 CU。这不是通货膨胀——这是对可用性的奖励。可靠在线的节点对网络更有价值。

### 声誉

每个节点都有一个介于 0.0 到 1.0 之间的声誉分数：

- 新节点从 0.5 开始
- 在线时间和成功的交易会增加声誉
- 断开连接和失败的交易会降低声誉
- 更高的声誉 → 更高的收益率，调度中的优先级更高

## 结算与外部桥接

### 核心规则

**协议以 CU 结算。** 转换为其他任何东西都是集成层面的考量。

### 结算报表

操作员可以导出任何时间窗口的可审计交易历史：

```
forge settle --hours 24 --price 0.05 --out settlement.json
```

报表包括：总收益 CU、总支出 CU、净 CU、交易计数，以及可选的每 CU 参考价格。

### 桥接架构

```
第 0 层: Forge 协议
  → CU 核算、交易、定价

第 1 层: 结算报表
  → 可导出的交易历史
  → 参考汇率

第 2 层: 外部桥接 (可选)
  → CU ↔ BTC (Lightning)
  → CU ↔ 稳定币
  → CU ↔ 法币
```

桥接层位于协议之外。不同的操作员可以使用不同的桥接。即使外部流动性为零，协议仍然有用。

### Lightning 桥接

对于希望对比特币结算的操作员：

```bash
forge settle --hours 24 --pay
```

这将根据净收益 CU，按配置的汇率（默认：每 CU 10 msats）创建一个 BOLT11 Lightning 发票。

## 代理主导的预算管理

### 愿景

传统模式：人类决定 → 人类支付 → AI 执行
Forge 模式：政策允许代理 → 代理检查预算 → 代理支出 CU → 代理执行

### API

```
GET /v1/forge/balance   → CU 余额、贡献、消耗、声誉
GET /v1/forge/pricing   → 市场价格，每 100/1000 token 的成本估算
```

代理可以：
1. 在发出请求前检查其余额
2. 根据当前市场价格估算推理成本
3. 决定请求是否值得花费 CU
4. 自动执行并支付

人类主管设置预算政策。代理在这些限制内自主运行。

### 自我强化循环

```
代理 (小型, 手机端)
  → 通过借出空闲计算赚取 CU
  → 花费 CU 购买大型模型访问权限
  → 变得更聪明
  → 做出更好的经济决策
  → 赚取更多 CU
  → 访问更大型的模型
  → ...
```

这是一种可能的应用模式。协议提供市场；代理提供策略。

## 为什么这不是 Web3

大多数 Web3 项目在丰富的数字商品之上创造人为稀缺性（代币）。Forge 则相反：

- **计算资源确实稀缺** — 它需要真实的电力、真实的硅片、真实的时间
- **CU 不是投机性的** — 它代表经过验证的工作，而不是对未来的赌注
- **没有 ICO，没有代币销售，没有治理代币** — CU 是通过工作赚取的
- **无需区块链** — 双边签名和 gossip 即可
- **没有智能合约** — 协议本身就是合同

价值是由物理规律制造的，而不是由共识制造的。
