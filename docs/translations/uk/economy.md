# Forge — Економічна модель

## Обчислювальний стандарт (Compute Standard)

Кожна грошова система підкріплена рідкістю:

| Епоха | Стандарт | Забезпечення |
|-----|----------|---------|
| Античність | Золото/Срібло | Геологічна рідкість |
| 1944–1971 | Бреттон-Вудс | USD, прив'язаний до золота |
| 1971–сьогодні | Петродолар | Попит на нафту + військова міць |
| 2009–сьогодні | Bitcoin | Електроенергія, спалена на SHA-256 |
| **Forge** | **Обчислювальний стандарт** | **Енергія, витрачена на корисний інференс** |

Forge впроваджує Обчислювальний Стандарт: одиниця цінності підкріплена реальним споживанням енергії для виконання корисних обчислень. На відміну від Proof of Work у Bitcoin, кожен джоуль, витрачений у Forge, створює реальний інтелект.

## CU: Нативна валюта

### Що таке CU

**1 CU = 1 мільярд FLOPs верифікованої роботи з інференсу.**

CU — це не криптовалюта. Це не токен у блокчейні. Це одиниця обліку, що представляє реальні виконані обчислення. CU має цінність, тому що це право на майбутні обчислення — якщо ви заробили CU, обслуговуючи інференс, ви можете витратити їх, щоб отримати інференс.

### Чому CU, а не Bitcoin

| Властивість | CU | Bitcoin |
|----------|-----|---------|
| **Забезпечення вартості** | Корисні обчислення (внутрішнє) | Хешування (штучне) |
| **Швидкість розрахунку** | Миттєво (локальний леджер) | Секунди до хвилин (Lightning/ланцюг) |
| **Вартість транзакції** | Нуль | Комісії каналів, комісії мережі |
| **Зовнішня залежність** | Немає | Стан мережі Bitcoin |
| **Квантовий ризик** | Немає (без крипто-загадок) | SHA-256 / ECDSA вразливі |
| **Генерація доходу** | Так (вільне залізо заробляє CU) | Ні (BTC у гаманці нічого не виробляє) |

CU є **основним** розрахунковим засобом. Bitcoin, стейблкоїни та фіат — це опціональні **виходи (off-ramps)**, доступні через адаптери мостів поза протоколом.

### CU як продуктивний актив

```
Багатоквартирний будинок      Mac Mini на Forge
───────────────────         ──────────────────
Актив: будівля              Актив: обчислювальне залізо
Витрати: обслуговування     Витрати: електроенергія
Дохід: оренда               Дохід: CU від інференсу
Прибуток: оренда - обсл.    Прибуток: зароблені CU - електр.
Простій = втрачений дохід   Простій = змарнований потенціал
```

Обчислювальний пристрій на Forge — це не як Bitcoin у гаманці (статична цінність, без доходу). Це як нерухомість в оренді — генерує дохід через корисну роботу.

## Модель транзакцій

### Виконання операції

Кожен інференс створює операцію (trade) між двома сторонами:

```rust
pub struct TradeRecord {
    pub provider: NodeId,       // Хто виконав інференс
    pub consumer: NodeId,       // Хто запитав його
    pub cu_amount: u64,         // Передано CU
    pub tokens_processed: u64,  // Виконана робота
    pub timestamp: u64,
    pub model_id: String,
}
```

Операція записується обома сторонами. У поточній реалізації кожен вузол веде локальний леджер. Цільова реалізація додає подвійні підписи та gossip-синхронізацію.

### Динамічне ціноутворення

Ціни CU коливаються залежно від місцевого попиту та пропозиції:

```
effective_price = base_cu_per_token × demand_factor / supply_factor
```

- **Більше вільних вузлів** → supply_factor зростає → ціна падає
- **Більше запитів інференсу** → demand_factor зростає → ціна зростає
- Кожен вузол спостерігає власні ринкові умови. Глобальної книги ордерів немає.

### Безкоштовний рівень

Нові вузли без історії внесків отримують 1,000 CU. Це дозволяє будь-кому почати використовувати мережу негайно. Безкоштовний рівень витрачається з першого запиту і не поновлюється.

Захист від атак Sybil: якщо в леджері з'явилося більше 100 невідомих вузлів без внесків, нові запити безкоштовного рівня відхиляються.

## Доказ корисної роботи (Proof of Useful Work)

### Концепція

Proof of Work у Bitcoin: "Я спалив електроенергію, обчислюючи хеші SHA-256. Ось nonce, який це доводить."

Forge Proof of Useful Work: "Я спалив електроенергію, виконуючи інференс LLM. Ось відповідь, і ось підпис споживача, який підтверджує її отримання."

Ключова відмінність: доказ Bitcoin створюється самостійно (будь-який майнер може створити валідний хеш). Доказ Forge потребує **контрагента** — того, кому справді був потрібен цей інференс. Ви не можете підробити попит.

### Протокол верифікації (ціль)

```
1. Споживач надсилає InferenceRequest Провайдеру
2. Провайдер виконує інференс, стрімить токени назад
3. Споживач отримує токени, обчислює хеш відповіді
4. Обидві сторони підписують TradeRecord:
   - Провайдер підписує: "Я обчислив це"
   - Споживач підписує: "Я отримав це"
5. TradeRecord з подвійним підписом розсилається через gossip у мережу
6. Будь-який вузол може перевірити обидва підписи
```

Вузол не може роздути свій баланс CU без співпраці з контрагентом. Змова можлива, але економічно ірраціональна — споживач у змові нічого не отримує від підпису фейкових операцій.

### Поточна реалізація

Поточна референсна реалізація використовує локальні леджери з захистом цілісності HMAC-SHA256. Подвійні підписи та gossip — наступний крок.

## Дохідність та репутація

### Дохідність (Yield)

Вузли, які залишаються онлайн та надають обчислення, заробляють дохід:

```
yield_cu = contributed_cu × 0.001 × reputation × uptime_hours
```

При репутації 1.0 вузол із 10,000 внесених CU заробляє 80 CU за 8-годинну ніч. Це не інфляція — це винагорода за доступність. Вузли, які надійно працюють в мережі, цінніші для неї.

### Репутація

Кожен вузол має показник репутації від 0.0 до 1.0:

- Нові вузли починають з 0.5
- Аптайм та успішні операції підвищують репутацію
- Відключення та невдалі операції знижують репутацію
- Вища репутація → вища ставка доходу, пріоритет у плануванні

## Розрахунки та зовнішні мости

### Основне правило

**Протокол розраховується в CU.** Конвертація в щось інше — це питання інтеграції.

### Звіти про розрахунки

Оператори можуть експортувати історію операцій для аудиту за будь-який період:

```
forge settle --hours 24 --price 0.05 --out settlement.json
```

Звіт включає: зароблено брутто CU, витрачено брутто CU, нетто CU, кількість операцій та опціональну референсну ціну за CU.

### Архітектура мостів

```
Шар 0: Протокол Forge
  → Облік CU, операції, ціни

Шар 1: Звіт про розрахунки
  → Історія операцій для експорту
  → Референсний обмінний курс

Шар 2: Зовнішній міст (опціонально)
  → CU ↔ BTC (Lightning)
  → CU ↔ stablecoin
  → CU ↔ фіат
```

Шар мостів знаходиться поза протоколом. Різні оператори можуть використовувати різні мости. Протокол залишається корисним навіть за нульової зовнішньої ліквідності.

### Міст Lightning

Для операторів, які хочуть розрахунків у Bitcoin:

```bash
forge settle --hours 24 --pay
```

Це створює Lightning-інвойс BOLT11 на суму зароблених нетто CU, конвертовану за встановленим курсом (за замовчуванням: 10 msats за 1 CU).

## Бюджети, керовані агентами

### Візія

Традиційно: Людина вирішує → Людина платить → ШІ виконує
Forge: Політика дозволяє агенту → Агент перевіряє бюджет → Агент витрачає CU → Агент виконує

### API

```
GET /v1/forge/balance   → баланс CU, внесок, споживання, репутація
GET /v1/forge/pricing   → ринкова ціна, оцінка вартості за 100/1000 токенів
```

Агент може:
1. Перевірити свій баланс перед запитом
2. Оцінити вартість інференсу за поточними ринковими цінами
3. Вирішити, чи вартий запит вартості в CU
4. Виконати та оплатити автоматично

Люди-куратори встановлюють політики бюджету. Агенти діють автономно в цих межах.

### Цикл самопідкріплення

```
Агент (малий, телефон)
  → заробляє CU, позичаючи вільні обчислення
  → витрачає CU на доступ до більшої моделі
  → стає розумнішим
  → приймає кращі економічні рішення
  → заробляє більше CU
  → отримує доступ до ще більших моделей
  → ...
```

Це можливий патерн застосування. Протокол забезпечує ринок; агенти забезпечують стратегію.

## Чому це не Web3

Більшість Web3-проектів створюють штучну рідкість (токени) поверх надлишкових цифрових благ. Forge робить навпаки:

- **Обчислення справді рідкісні** — вони потребують реальної енергії, кремнію, часу
- **CU не є спекулятивним** — він представляє верифіковану роботу, а не ставку на майбутнє поширення
- **Ніяких ICO, продажу токенів, токенів управління** — CU заробляється працею
- **Блокчейн не потрібен** — двосторонніх підписів та gossip достатньо
- **Ніяких смарт-контрактів** — протокол сам є контрактом

Цінність виробляється фізикою, а не консенсусом.
