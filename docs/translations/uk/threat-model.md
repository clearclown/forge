# Forge — Модель загроз

## Цілі безпеки

1. **Конфіденційність при передачі**: пасивні спостерігачі та реле не повинні мати можливості читати промпти або відповіді.
2. **Автентифіковані піри**: кожне пряме з'єднання має бути прив'язане до криптографічної ідентичності вузла.
3. **Обмежена довіра**: система має чітко визначати, які вузли бачать відкритий текст, а які — лише проміжний стан.
4. **Доступність**: збої окремих пірів повинні призводити до деградації сервісу, а не до прихованого пошкодження даних.

## Шифрування

### Транспортний шар
- Усі з'єднання використовують QUIC з TLS 1.3 (через Iroh).
- Додаткове рукостискання Noise Protocol (патерн XX) для автентифікації пірів.
- Результат: симетричне шифрування ChaCha20-Poly1305 з прямою секретністю (forward secrecy).
- Ефемерні ключі для кожної сесії — компрометація однієї сесії не впливає на інші.

### Ідентичність
- Кожен вузол має постійну пару ключів Ed25519.
- Генерується при першому запуску, зберігається в системному сховищі ключів.
- Node ID = хеш публічного ключа.
- Без центрального центру сертифікації — модель мережі довіри (web-of-trust).

### Що шифрується
| Дані | Зашифровано | Примітки |
|---|---|---|
| Текст промпту | Так | Зашифровано при передачі; видно сіду в поточному референсному потоці |
| Стрімінг тексту | Так | Зашифровано при передачі; видно сіду, який його згенерував |
| Тензори активації | Планується | Актуально після впровадження розділеного інференсу (split inference) |
| Контрольні повідомлення | Так | Усі повідомлення протоколу всередині QUIC |
| Можливості пірів | Так | Обмін через зашифрований канал |

## Аналіз загроз

### T1: Зловмисний сід-вузол (поточний референсний потік)
**Загроза**: Оператор сіда читає промпт або відповідь, оскільки воркер надсилає текст промпту сіду, а сід запускає повну модель.

**Поточний статус**: Це явна межа довіри, а не вирішена властивість безпеки.

**Заходи протидії**:
- підключайте воркерів лише до сідів, яким ви довіряєте відкриті промпти
- тримайте транспорт зашифрованим, щоб реле та пасивні спостерігачі не могли читати вміст
- рухайтеся до розділеного інференсу, щоб піри на проміжних стадіях не отримували відкриті промпти

### T2: Зловмисний вузол пайплайну (цільовий потік розділеного інференсу)
**Загроза**: Вузол у майбутньому пайплайні намагається вилучити промпт або відповідь із проміжних активацій.

**Цільовий захід**: Вузол на стадії пайплайну k має бачити лише вихідний тензор активації шару k-1 і створювати тензор активації для шару k. Він не повинен отримувати оригінальний текст промпту.

**Залишковий ризик**: Тензори активації можуть викривати інформацію про вхідні дані. Диференційна приватність, надмірність та атестація залишаються завданнями на майбутнє.

### T3: Атака Sybil
**Загроза**: Зловмисник створює багато фейкових вузлів, щоб домінувати в пайплайні.

**Заходи протидії**:
- Система репутації на основі спостережуваної поведінки (аптайм, коректність обчислень)
- Нові вузли починають з низькою репутацією та обмеженими позиціями в пайплайні
- Критичні шари (перший та останній) переважно призначаються вузлам з високою репутацією
- Обмеження швидкості (rate limiting) приєднання нових вузлів з того самого діапазону IP

### T4: Візантійський інференс (Byzantine Inference)
**Загроза**: Зловмисний вузол повертає некоректні тензори активації.

**Захід (MVP)**: Прийняти ризик. Для більшості випадків використання користувач може помітити некоректний результат інференсу.

**Заходи (майбутнє)**:
- Надмірні обчислення на критичних шарах (2 вузли обчислюють ті самі шари, порівняння)
- Верифіковані обчислення з використанням атестації TEE (Apple Silicon Secure Enclave)
- Статистичне виявлення аномалій у розподілах тензорів активації

### T5: Аналіз трафіку
**Загроза**: Спостерігач відстежує зашифровані патерни трафіку, щоб зробити висновки про використання.

**Заходи протидії**:
- QUIC мультиплексує весь зв'язок через одне з'єднання
- Поточний трафік seed/worker все ще викриває грубі метадані про час запиту та довжину відповіді
- Додавання заповнювача (padding) до контрольних повідомлень до постійного розміру (опціонально, не в MVP)

### T6: Компрометація реле-сервера
**Загроза**: Бутстрап реле-сервери скомпрометовані.

**Вплив**: Мінімальний. Реле-сервери лише допомагають встановити з'єднання. Вони бачать:
- Які Node ID підключаються (метадані)
- Зашифровані пакети QUIC (не можуть розшифрувати)
- Вони НЕ бачать розшифровані промпти або відповіді

**Заходи протидії**: Кілька незалежних операторів реле. Мережа продовжує працювати без реле після заповнення DHT.

### T7: Отруєння моделі (Model Poisoning)
**Загроза**: Вузол обслуговує модифіковану модель GGUF з бекдорами у вагах.

**Заходи протидії**:
- Файли моделей перевіряються за хешем SHA-256 проти відомих маніфестів
- Маніфести моделей поширюються через DHT з підписами видавців моделей
- Вузли завантажують моделі лише з перевірених джерел (хеші HuggingFace)

### T8: Відмова в обслуговуванні (DoS)
**Загроза**: Вузли приєднуються, а потім перестають відповідати, перериваючи інференс.

**Заходи протидії**:
- Таймаут хартбітів та ребалансування — це цільові властивості рантайму, а не повні гарантії поточної реалізації
- Локальний фолбек є ціллю дизайну для майбутніх клієнтів розділеного інференсу
- Штраф до репутації для вузлів, які часто відключаються
- Граціозна деградація є основним принципом дизайну
- Вхідні запити інференсу обмежені валідацією рантайму та фіксованим лімітом одночасного виконання на сіді
- Дублікати `msg_id` протоколу від одного й того самого піра відкидаються всередині обмеженого вікна повторів

### T9: Відкриття адміністративного API
**Загроза**: Оператор прив'язує локальне HTTP API до публічного інтерфейсу без захисту, відкриваючи `/status`, `/topology`, `/settlement` або `/chat`.

**Заходи протидії в поточної реалізації**:
- демон за замовчуванням прив'язує HTTP API до `127.0.0.1`
- оператори все ще можуть відкрити його навмисно через `--bind 0.0.0.0`
- відкриті адміністративні шляхи можна захистити bearer-токеном через `--api-token`
- тіла JSON-запитів обмежені за розміром перед десеріалізацією, щоб зменшити зловживання пам'яттю на `/chat`

**Залишковий ризик**: Автентифікація через bearer-токен — це контроль оператора, а не взаємний TLS. Якщо токен просочиться, API слід вважати скомпрометованим до моменту ротації.

## Ієрархія довіри

```
Найбільша довіра: Власний пристрій (телефон, ноутбук)
                 ↓
Довірені:        Власні пристрої в локальній мережі (Mac Mini вдома)
                 ↓
Напівдовірені:   Піри у WAN з високою репутацією (місяці аптайму)
                 ↓
Недовірені:      Нові піри у WAN (щойно приєдналися, без історії)
```

Призначення шарів має слідувати цій ієрархії після появи розділеного інференсу:
- Перший та останній шари (найчутливіші — бачать вхідні ембеддінги та вихідні логіти) → власні пристрої
- Середні шари (бачать лише проміжні активації) → можуть бути призначені напівдовіреним або недовіреним пірам

## Гарантії приватності

**Що Forge гарантує сьогодні:**
- промпти та відповіді шифруються при передачі між прямо підключеними пірами
- реле та пасивні спостерігачі мережі не бачать розшифрований вміст промптів або відповідей
- у шляху даних немає обов'язкового центрального сервера
- поточна межа довіри seed/worker є явною

**Чого Forge не гарантує сьогодні:**
- що сід не може прочитати промпт або відповідь
- що розділений інференс приховує відкритий текст від усіх віддалених провайдерів обчислень
- що некоректний віддалений інференс виявляється автоматично

**Що Forge прагне гарантувати пізніше:**
- піри на середніх стадіях не отримують відкриті тексти промптів
- тензори активації шифруються при передачі між стадіями пайплайну
- видимість промпту зведена до мінімального набору довірених вузлів на межі

Ці пізніші гарантії залежать від реалізації справжнього розділеного інференсу. До того часу Forge слід описувати як зашифрований віддалений інференс із чесною межею довіри.

## Економічні загрози

### T10: Підробка CU

**Загроза**: Вузол заявляє CU, які він не заробив, фабрикуючи записи TradeRecord.

**Поточний захід**: Локальний леджер з цілісністю HMAC-SHA256 запобігає втручанню на рівні файлів. Однак оператор вузла все ще може записувати довільні операції у свій леджер.

**Цільовий захід**: Протокол подвійного підпису. Кожен TradeRecord має бути підписаний і провайдером, і споживачем. Вузол не може нарахувати собі CU без підпису контрагента. Gossip-синхронізація означає, що інші вузли можуть перевірити обидва підписи незалежно.

**Залишковий ризик**: Змова між провайдером та споживачем для створення фейкових операцій. Це економічно ірраціонально — споживач у змові нічого не отримує. Статистичне виявлення аномалій обсягу та частоти операцій може позначати підозрілі патерни.

### T11: Зловживання безкоштовним рівнем (Sybil)

**Загроза**: Зловмисник створює багато нових NodeId, щоб багаторазово використовувати безкоштовні 1,000 CU.

**Поточний захід**: Якщо в леджері існує більше 100 невідомих вузлів (внесок = 0, споживання > 0), нові запити безкоштовного рівня відхиляються. Кожен NodeId — це пара ключів Ed25519 — дешево створити, але можна відстежити.

**Цільовий захід**: Proof of Work при реєстрації вузла (невеликі обчислювальні витрати на створення нової ідентичності) або вхід на основі стейкінгу (нові вузли мають внести обчислення перед споживанням).

### T12: Розходження леджерів

**Загроза**: Різні вузли мають несумісні погляди на ті самі операції, що призводить до економічної неузгодженості.

**Поточний захід**: Кожен вузол підтримує власний локальний погляд. Гарантії узгодженості між вузлами немає.

**Цільовий захід**: Синхронізовані через gossip записи TradeRecord з подвійним підписом. Обидві сторони створюють однакові підписані записи. Будь-який вузол, що отримує gossip-оновлення, може перевірити підписи та відхилити невідповідності. Періодичне закріплення підсумків у Bitcoin (OP_RETURN) забезпечує опціональний незмінний аудиторський слід.

### T13: Маніпулювання ринком

**Загроза**: Вузол штучно роздуває фактори попиту або пропозиції для маніпулювання цінами.

**Поточний захід**: Ринкова ціна обчислюється локально на основі власних спостережень кожного вузла. Жоден окремий вузол не може змусити інший прийняти свою ціну.

**Цільовий захід**: Цінові сигнали на основі gossip, зважені за репутацією. Спостереження вузлів з високою репутацією мають більшу вагу. Нові вузли або вузли з низькою репутацією не можуть суттєво впливати на ціни в усій мережі.

### T14: Атака на якість інференсу

**Загроза**: Провайдер повертає низькоякісний або обрізаний інференс, щоб заробити CU, не виконуючи повних обчислень.

**Поточний захід**: Прийняти ризик. Для більшості випадків явно неправильні відповіді помітні споживачу.

**Цільовий захід**: Верифікація якості на стороні споживача. Споживач може повторно запустити невелику вибірку токенів локально, щоб перевірити відповідність виводу провайдера. Штраф до репутації для провайдерів, чиї результати не пройшли перевірку.
