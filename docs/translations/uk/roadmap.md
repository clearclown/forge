# Forge — Роадмап

## Фаза 1: Локальний інференс ✅

- `forge-core`: Система типів (NodeId, LayerRange, ModelManifest, PeerCapability)
- `forge-infer`: рушій llama.cpp, завантажувач GGUF, стрімінгова генерація токенів
- `forge-node`: HTTP API (/chat, /chat/stream, /health)
- `forge-cli`: команда `forge chat` з автозавантаженням моделей

## Фаза 2: Протокол P2P ✅

- `forge-net`: транспорт Iroh, шифрування Noise, з'єднання пірів
- `forge-proto`: 14 типів повідомлень протоколу (bincode + префікс довжини)
- `forge-node`: пайплайн Seed/Worker, запит/відповідь інференсу
- Інтеграційні тести: 2 вузли обмінюються Hello + кількома повідомленнями

## Фаза 3: Віддалений інференс + Леджер оператора ✅

- `forge-ledger`: облік CU, виконання операцій, репутація, дохідність, ринкове ціноутворення
- `forge-node`: леджер інтегрований у пайплайн інференсу
- Перевірка балансу CU перед інференсом
- Запис операцій після завершення
- Цілісність леджера HMAC-SHA256

## Фаза 4: Економічне API ✅

- API, сумісне з OpenAI: `POST /v1/chat/completions`, `GET /v1/models`
- Метрика CU: кожен інференс записує операцію з розширенням `x_forge`
- Ендпоінти бюджету агента: `GET /v1/forge/balance`, `GET /v1/forge/pricing`
- Міст розрахунків CU→Lightning: `forge settle --pay`
- Автоматичне знаходження моделей на HF Hub
- Коректне завершення роботи через Ctrl-C зі збереженням леджера

## Фаза 5: Інтеграція форка mesh-llm (наступна)

**Ціль:** Замінити шар інференсу Forge перевіреним розподіленим рушієм mesh-llm.

| Результат | Опис |
|---|---|
| Форк mesh-llm | Створити forge як форк mesh-llm з економічним шаром |
| Інтеграція forge-ledger | Підключити запис CU до пайплайну інференсу mesh-llm |
| Збереження економічного API | Залишити ендпоінти /v1/forge/* у новому коді |
| Розширення веб-консолі | Додати візуалізацію балансу CU та операцій у консоль mesh-llm |
| Pipeline + MoE | Успадкувати паралелізм пайплайну та шардинг експертів mesh-llm |
| Nostr discovery | Успадкувати пошук публічних мереж через Nostr |
| CREDITS.md | Документувати авторство mesh-llm |

## Фаза 6: Доказ корисної роботи

**Ціль:** Зробити заяви про зароблені CU верифікованими в мережі.

| Результат | Опис |
|---|---|
| Протокол подвійного підпису | Провайдер та споживач підписують кожну TradeRecord |
| Gossip sync | Підписані операції поширюються по всій мережі |
| Виявлення шахрайства | Відхилення непідписаних або невідповідних операцій |
| Gossip репутації | Обмін оцінками репутації між пірами |
| Опір змовам | Статистичне виявлення аномалій у моделях торгівлі |

## Фаза 7: Зовнішні мости

**Ціль:** Дозволити операторам конвертувати CU у зовнішню цінність.

| Результат | Опис |
|---|---|
| Міст Lightning | Автоматизовані розрахунки CU→sats через LDK |
| Адаптер Stablecoin | Конвертація CU→USDC/USDT |
| Інтерфейс адаптера фіату | Специфікація для розрахунків банківськими переказами |
| Сервіс курсів обміну | Публічні фіди курсів CU/BTC та CU/USD |
| Анкоринг у Bitcoin | Опціонально: періодичний корінь Меркла → OP_RETURN для незмінного аудиту |

## Фаза 8: Автономна економіка агентів

**Ціль:** Дозволити ШІ-агентам самостійно керувати своїм життєвим циклом обчислень.

| Результат | Опис |
|---|---|
| Політики бюджету | Встановлені людиною ліміти витрат на агента |
| Автономна торгівля | Агент вирішує, коли купувати/продавати обчислення |
| Мультимодельний роутинг | Агент обирає модель на основі компромісу ціна/якість |
| Самопідкріплення | Агент заробляє CU → купує доступ до кращої моделі → заробляє більше |
| Міжагентська економіка | Агенти торгують спеціалізованими обчисленнями (код-модель проти чат-моделі) |

## Довгострокові перспективи

| Віха | Опис |
|---|---|
| Реліз SDK | forge-node як Rust-бібліотека зі стабільним API |
| Протокол v2 | Уроки v1, еволюція зі зворотною сумісністю |
| Крос-архітектурність | Підтримка NVIDIA GPU, AMD ROCm, RISC-V (через mesh-llm) |
| Федеративне навчання | Розподілений fine-tuning, а не тільки інференс |
| Обчислювальні деривативи | Форвардні контракти на майбутні обчислювальні потужності |

> Протокол — це платформа. Обчислення — це валюта.
