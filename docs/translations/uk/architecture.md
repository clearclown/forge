# Forge — Архітектура

## Огляд

Forge — це двошарова система: **інференс** та **економіка**.

Шар інференсу займається розподілом моделей, mesh-мережами та обслуговуванням API. Він побудований на [mesh-llm](https://github.com/michaelneale/mesh-llm).

Економічний шар займається обліком CU, записом операцій, ціноутворенням та бюджетами агентів. Це оригінальний внесок Forge.

```
┌─────────────────────────────────────────────────┐
│  Межа SDK / Інтеграції                          │
│  Будь-який клієнт може вбудувати forge-node     │
│  Сторонні агенти, дашборди, адаптери            │
└──────────────────┬──────────────────────────────┘
                   │
┌──────────────────▼──────────────────────────────┐
│  Економічний шар (Оригінальний Forge)           │
│                                                  │
│  ┌──────────────┐ ┌──────────┐ ┌─────────────┐ │
│  │ forge-ledger │ │ pricing  │ │ agent       │ │
│  │ операції CU  │ │ попит/   │ │ бюджети     │ │
│  │ репутація    │ │ пропозиція│ │ /v1/forge/* │ │
│  │ дохідність   │ │          │ │             │ │
│  └──────────────┘ └──────────┘ └─────────────┘ │
│                                                  │
│  ┌──────────────┐ ┌──────────────────────────┐  │
│  │ forge-verify │ │ forge-bridge (опція)     │  │
│  │ двост. підпис│ │ CU ↔ BTC Lightning      │  │
│  │ gossip sync  │ │ CU ↔ stablecoin         │  │
│  └──────────────┘ └──────────────────────────┘  │
└──────────────────┬──────────────────────────────┘
                   │
┌──────────────────▼──────────────────────────────┐
│  Шар інференсу (на базі mesh-llm)               │
│                                                  │
│  ┌────────────┐ ┌───────────┐ ┌──────────────┐ │
│  │ iroh mesh  │ │ llama.cpp │ │ OpenAI API   │ │
│  │ QUIC+Noise │ │ pipeline  │ │ /v1/chat/    │ │
│  │ Nostr disc │ │ MoE shard │ │ completions  │ │
│  └────────────┘ └───────────┘ └──────────────┘ │
```

## Шар інференсу (mesh-llm)

Шар інференсу відповідає за:

- **Mesh-мережі**: QUIC-з'єднання на базі iroh з шифруванням Noise
- **Пошук пірів**: Nostr реле для публічних мереж, mDNS для локальних мереж
- **Розподіл моделей**: Паралелізм пайплайну для щільних моделей, шардинг експертів для MoE
- **Виконання інференсу**: llama.cpp через підпроцеси llama-server та rpc-server
- **Обслуговування API**: сумісні з OpenAI `/v1/chat/completions` та `/v1/models`

Forge успадковує все це від mesh-llm. Шар інференсу нічого не знає про CU, операції або ціноутворення.

## Економічний шар (Forge)

Економічний шар знаходиться над інференсом і відповідає за:

### forge-ledger — економічний рушій

```rust
pub struct ComputeLedger {
    balances: HashMap<NodeId, NodeBalance>,
    work_log: Vec<WorkUnit>,
    trade_log: Vec<TradeRecord>,
    price: MarketPrice,
}
```

Основні обов'язки:
- Відстеження балансу CU кожного вузла (внесок, споживання, резерв)
- Запис кожної операції інференсу (провайдер, споживач, сума CU, токени)
- Обчислення динамічних ринкових цін на основі попиту та пропозиції
- Нарахування доходу вузлам-учасникам
- Експорт звітів для розрахунків через зовнішні мости
- Збереження знімків (snapshots) на диск із захистом цілісності HMAC-SHA256

### forge-verify — Доказ корисної роботи (ціль)

Забезпечує легітимність вимог CU:
- Протокол подвійного підпису: і провайдер, і споживач підписують кожну TradeRecord
- Gossip-синхронізація: підписані операції поширюються по мережі
- Верифікація: будь-який вузол може перевірити обидва підписи
- Виявлення шахрайства: невідповідні або непідписані операції відхиляються

### forge-bridge — Зовнішні розрахунки (опціонально)

Конвертує CU у зовнішню вартість для операторів, яким це потрібно:
- Bitcoin Lightning: CU → msats через настроюваний обмінний курс
- Stablecoin: CU → USDC/USDT через адаптер
- Фіат: CU → банківський переказ через панель оператора

Шар мостів знаходиться поза ядром протоколу. Різні оператори можуть використовувати різні мости.

### Склад API

| Шлях | Шар | Опис |
|-------|-------|-------------|
| `POST /v1/chat/completions` | Інференс + Економіка | Запуск інференсу, запис операції CU |
| `GET /v1/models` | Інференс | Список завантажених моделей |
| `GET /v1/forge/balance` | Економіка | Баланс CU, репутація |
| `GET /v1/forge/pricing` | Економіка | Ринкова ціна, оцінка вартості |
| `GET /status` | Економіка | Ринкова ціна, статистика мережі, останні операції |
| `GET /topology` | Інференс | Маніфест моделі, піри, план шардингу |
| `GET /settlement` | Економіка | Історія операцій для експорту |
| `GET /health` | Інференс | Базова перевірка працездатності |

## Потік даних

### Інференс з обліком CU

```
Споживач надсилає запит
    ↓
API отримує POST /v1/chat/completions
    ↓
Леджер перевіряє: can_afford(consumer, estimated_cost)?
    ↓ так
Шар інференсу виконує (llama-server / rpc-server)
    ↓
Токени стрімяться назад до споживача
    ↓
Леджер записує операцію:
  - provider.contributed += cu_cost
  - consumer.consumed += cu_cost
  - trade_log.push(TradeRecord)
    ↓
Відповідь включає x_forge: { cu_cost, effective_balance }
```

### Експорт розрахунків

```
Оператор запускає: forge settle --hours 24
    ↓
API читає trade_log за вказаний період
    ↓
Агрегує по кожному вузлу: зароблене брутто, витрачене брутто, нетто CU
    ↓
Експортує звіт JSON з опціональною референсною ціною
    ↓
Оператор використовує адаптер моста для конвертації нетто CU в BTC/фіат
```

## Модель безпеки

```
Шар 0: Bitcoin mainchain    ← Опціональне закріплення (майбутнє)
Шар 1: Подвійні підписи     ← Провайдер + споживач підписують кожну операцію
Шар 2: HMAC-SHA256 леджер   ← Локальний захист цілісності
Шар 3: iroh (QUIC + Noise)  ← Шифрування транспорту
Шар 4: Виконання інференсу  ← Модель працює локально на провайдері
```

Кожен шар захищає від різних загроз:
- Шар 4: Цілісність моделі (верифікація хешу GGUF)
- Шар 3: Конфіденційність транспорту (прослуховування)
- Шар 2: Локальне втручання (зміна файлів)
- Шар 1: Мережеве шахрайство (фейкові заяви CU)
- Шар 0: Історична незмінність (опціональний якір Bitcoin)

## Залежності крейтів

```
forge-core ← спільні типи (NodeId, CU, Config)
    ↑
forge-ledger ← економічний рушій (операції, ціни, дохід)
    ↑
forge-lightning ← зовнішній міст (LDK гаманець, CU↔sats)
    ↑
forge-node ← оркестратор (HTTP API, pipeline, інтеграція леджера)
    ↑
forge-cli ← еталонний CLI (chat, seed, worker, settle)

forge-net ← P2P транспорт (iroh, QUIC, Noise, mDNS)
forge-proto ← повідомлення (bincode, 14 типів корисного навантаження)
forge-infer ← рушій інференсу (llama.cpp, GGUF завантажувач)
forge-shard ← планувальник топології (призначення шарів, ребалансування)
```
