# Forge — الهندسة المعمارية (Architecture)

## نظرة عامة (Overview)

نظام Forge هو نظام ذو طبقتين: **الاستنتاج (inference)** و **الاقتصاد (economy)**.

تتولى طبقة الاستنتاج مسؤولية توزيع النماذج، وشبكات المش (mesh networking)، وتقديم واجهة برمجة التطبيقات (API serving). وهي مبنية على [mesh-llm](https://github.com/michaelneale/mesh-llm).

أما الطبقة الاقتصادية فتتولى محاسبة وحدات الحوسبة (CU)، وتسجيل الصفقات، والتسعير، وميزانيات الوكلاء (agent budgets). هذا هو الابتكار الأصلي لـ Forge.

└──────────────────┬──────────────────────────────┘

│  │ forge-ledger │ │ التسعير   │ │ ميزانيات     │ │

## طبقة الاستنتاج (mesh-llm)

طبقة الاستنتاج مسؤولة عن:

- **شبكة المش (Mesh networking)**: اتصالات QUIC مبنية على iroh مع تشفير Noise.
- **اكتشاف الأقران (Peer discovery)**: مرحلات Nostr للمش العام، و mDNS للشبكات المحلية (LAN).
- **توزيع النماذج (Model distribution)**: توازي خط الأنابيب (Pipeline parallelism) للنماذج الكثيفة، وتجزئة الخبراء (expert sharding) لنماذج MoE.
- **تنفيذ الاستنتاج (Inference execution)**: llama.cpp عبر العمليات الفرعية llama-server و rpc-server.
- **تقديم واجهة برمجة التطبيقات (API serving)**: متوافقة مع OpenAI في المسارات `/v1/chat/completions` و `/v1/models`.

يرث Forge كل هذا من mesh-llm. طبقة الاستنتاج لا تعرف شيئاً عن وحدات الحوسبة (CU)، أو الصفقات، أو التسعير.

## الطبقة الاقتصادية (Forge)

تقع الطبقة الاقتصادية فوق طبقة الاستنتاج وهي مسؤولة عن:

### forge-ledger — المحرك الاقتصادي

```rust
pub struct ComputeLedger {
    balances: HashMap<NodeId, NodeBalance>,
    work_log: Vec<WorkUnit>,
    trade_log: Vec<TradeRecord>,
    price: MarketPrice,
}
```

المسؤوليات الأساسية:
- تتبع رصيد CU لكل عقدة (المساهمة، المستهلكة، المحجوزة).
- تسجيل كل صفقة استنتاج (المزود، المستهلك، كمية CU، التوكنات).
- حساب أسعار السوق الديناميكية بناءً على العرض والطلب.
- تطبيق العائد (Yield) على العقد المساهمة.
- تصدير بيانات التسوية للجسور خارج البروتوكول.
- حفظ لقطات البيانات (snapshots) على القرص مع حماية التكامل HMAC-SHA256.

### forge-verify — إثبات العمل المفيد (المستهدف)

يضمن شرعية مطالبات CU:
- بروتوكول التوقيع المزدوج (Dual-sign): يقوم كل من المزود والمستهلك بتوقيع كل سجل صفقة (TradeRecord).
- مزامنة Gossip: تنتشر الصفقات الموقعة عبر الشبكة.
- التحقق: يمكن لأي عقدة التحقق من كلا التوقيعين.
- كشف الاحتيال: يتم رفض الصفقات غير المتطابقة أو غير الموقعة.

### forge-bridge — التسوية الخارجية (اختياري)

يحول وحدات الحوسبة (CU) إلى قيمة خارجية للمشغلين الذين يحتاجون إليها:
- Bitcoin Lightning: تحويل CU ← msats عبر سعر صرف قابل للضبط.
- العملات المستقرة (Stablecoin): تحويل CU ← USDC/USDT عبر محول (adapter).
- العملات النقدية (Fiat): تحويل CU ← تحويل بنكي عبر لوحة تحكم المشغل.

طبقة الجسر تقع خارج البروتوكول الأساسي. يمكن للمشغلين المختلفين استخدام جسور مختلفة.

### واجهة برمجة التطبيقات (API Surface)

| المسار | الطبقة | الوصف |
|-------|-------|-------------|
| `POST /v1/chat/completions` | الاستنتاج + الاقتصاد | تشغيل الاستنتاج، تسجيل صفقة CU |
| `GET /v1/models` | الاستنتاج | قائمة النماذج المحملة |
| `GET /v1/forge/balance` | الاقتصاد | رصيد CU، السمعة |
| `GET /v1/forge/pricing` | الاقتصاد | سعر السوق، تقديرات التكلفة |
| `GET /status` | الاقتصاد | سعر السوق، إحصائيات الشبكة، الصفقات الأخيرة |
| `GET /topology` | الاستنتاج | بيان النموذج، الأقران، خطة التجزئة |
| `GET /settlement` | الاقتصاد | تاريخ الصفقات القابل للتصدير |
| `GET /health` | الاستنتاج | فحص الصحة الأساسي |

## تدفق البيانات (Data Flow)

### الاستنتاج مع محاسبة CU

```
المستهلك يرسل الطلب
    ↓
واجهة البرمجة تستلم POST /v1/chat/completions
    ↓
دفتر الحسابات (Ledger) يتحقق: هل يستطيع المستهلك الدفع (can_afford)؟
    ↓ نعم
طبقة الاستنتاج تنفذ (llama-server / rpc-server)
    ↓
التوكنات تتدفق عائدة للمستهلك
    ↓
دفتر الحسابات يسجل الصفقة:
  - المزود: provider.contributed += cu_cost
  - المستهلك: consumer.consumed += cu_cost
  - إضافة سجل: trade_log.push(TradeRecord)
    ↓
الاستجابة تتضمن x_forge: { cu_cost, effective_balance }
```

### تصدير التسوية

```
المشغل يشغل: forge settle --hours 24
    ↓
واجهة البرمجة تقرأ trade_log للفترة الزمنية
    ↓
تجميع البيانات لكل عقدة: إجمالي المكتسب، إجمالي المنفق، صافي CU
    ↓
تصدير بيان JSON مع سعر مرجعي اختياري
    ↓
المشغل يستخدم محول الجسر لتحويل صافي CU إلى BTC/نقد
```

## النموذج الأمني

```
الطبقة 0: بلوكشين بيتكوين     ← تثبيت اختياري (مستقبلاً)
الطبقة 1: التوقيعات المزدوجة  ← المزود والمستهلك يوقعان كل صفقة
الطبقة 2: دفتر HMAC-SHA256    ← حماية التكامل المحلي
الطبقة 3: iroh (QUIC + Noise) ← تشفير النقل
الطبقة 4: تنفيذ الاستنتاج      ← النموذج يعمل محلياً عند المزود
```

كل طبقة تحمي من تهديدات مختلفة:
- الطبقة 4: تكامل النموذج (التحقق من هاش GGUF).
- الطبقة 3: سرية النقل (التنصت).
- الطبقة 2: التلاعب المحلي (تعديل الملفات).
- الطبقة 1: الاحتيال في الشبكة (مطالبات CU وهمية).
- الطبقة 0: عدم قابلية التغيير التاريخي (تثبيت اختياري في بيتكوين).

## تبعات الصناديق (Crate Dependencies)

```
forge-core ← أنواع مشتركة (NodeId, CU, Config)
    ↑
forge-ledger ← المحرك الاقتصادي (الصفقات، التسعير، العائد)
    ↑
forge-lightning ← الجسر الخارجي (محفظة LDK، تحويل CU↔sats)
    ↑
forge-node ← المنسق (واجهة HTTP، خط الأنابيب، تكامل دفتر الحسابات)
    ↑
forge-cli ← واجهة السطر البرمجي المرجعية (chat, seed, worker, settle)

forge-net ← نقل P2P (iroh, QUIC, Noise, mDNS)
forge-proto ← رسائل الشبكة (bincode, 14 payload types)
forge-infer ← محرك الاستنتاج (llama.cpp, GGUF loader)
forge-shard ← مخطط الطوبولوجيا (تعيين الطبقات، إعادة التوازن)
```
