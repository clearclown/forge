# Forge — معماری (Architecture)

## بررسی اجمالی (Overview)

Forge یک سیستم دو لایه است: **استنتاج (Inference)** و **اقتصاد (Economy)**.

لایه استنتاج وظیفه توزیع مدل، شبکه مش (Mesh) و ارائه API را بر عهده دارد. این لایه بر پایه [mesh-llm](https://github.com/michaelneale/mesh-llm) ساخته شده است.

لایه اقتصاد وظیفه حسابداری CU، ثبت معاملات، قیمت‌گذاری و بودجه‌بندی عوامل را بر عهده دارد. این بخش، آورده اصلی و اختصاصی Forge است.

```
┌─────────────────────────────────────────────────┐
│  مرز SDK / یکپارچه‌سازی                            │
```

## لایه استنتاج (mesh-llm)

لایه استنتاج مسئول موارد زیر است:

- **شبکه مش (Mesh networking)**: اتصالات QUIC مبتنی بر iroh با رمزنگاری Noise
- **شناسایی همتایان (Peer discovery)**: رله‌های Nostr برای مش‌های عمومی، mDNS برای شبکه محلی (LAN)
- **توزیع مدل (Model distribution)**: موازی‌سازی پایپ‌لاین برای مدل‌های متراکم، شاردینگ متخصص (Expert sharding) برای MoE
- **اجرای استنتاج (Inference execution)**: llama.cpp از طریق زیر‌فرآیندهای llama-server و rpc-server
- **ارائه API**: نقاط انتهایی `/v1/chat/completions` و `/v1/models` سازگار با OpenAI

Forge تمام این موارد را از mesh-llm به ارث می‌برد. لایه استنتاج هیچ اطلاعی از CU، معاملات یا قیمت‌گذاری ندارد.

## لایه اقتصادی (Forge)

لایه اقتصادی بالای لایه استنتاج قرار دارد و مسئول موارد زیر است:

### forge-ledger — موتور اقتصادی

```rust
pub struct ComputeLedger {
    balances: HashMap<NodeId, NodeBalance>,
    work_log: Vec<WorkUnit>,
    trade_log: Vec<TradeRecord>,
    price: MarketPrice,
}
```

مسئولیت‌های اصلی:
- پیگیری موجودی CU هر نود (مشارکت شده، مصرف شده، رزرو شده)
- ثبت هر معامله استنتاج (ارائه‌دهنده، مصرف‌کننده، مقدار CU، توکن‌ها)
- محاسبه قیمت‌های پویای بازار بر اساس عرضه و تقاضا
- اعمال سوددهی (Yield) به نودهای مشارکت‌کننده
- خروجی گرفتن از صورت‌حساب‌های تسویه برای پل‌های خارج از پروتکل
- ذخیره اسنپ‌شات‌ها روی دیسک با تایید یکپارچگی HMAC-SHA256

### forge-verify — اثبات کار مفید (هدف)

اطمینان از قانونی بودن ادعاهای CU:
- پروتکل امضای دوگانه (Dual-sign): هم ارائه‌دهنده و هم مصرف‌کننده هر TradeRecord را امضا می‌کنند
- همگام‌سازی (Gossip sync): معاملات امضا شده در سراسر شبکه پخش می‌شوند
- تایید اعتبار: هر نودی می‌تواند هر دو امضا را اعتبار‌سنجی کند
- شناسایی تقلب: معاملات نامطابق یا بدون امضا رد می‌شوند

### forge-bridge — تسویه خارجی (اختیاری)

تبدیل CU به ارزش خارجی برای اپراتورهایی که به آن نیاز دارند:
- Bitcoin Lightning: تبدیل CU → msats از طریق نرخ تبادل قابل تنظیم
- Stablecoin: تبدیل CU → USDC/USDT از طریق آداپتور
- Fiat: تبدیل CU → انتقال بانکی از طریق داشبورد اپراتور

لایه پل (Bridge) خارج از پروتکل اصلی قرار دارد. اپراتورهای مختلف می‌توانند از پل‌های متفاوتی استفاده کنند.

### سطح API

| مسیر | لایه | توضیحات |
|-------|-------|-------------|
| `POST /v1/chat/completions` | استنتاج + اقتصاد | اجرای استنتاج، ثبت معامله CU |
| `GET /v1/models` | استنتاج | لیست مدل‌های بارگذاری شده |
| `GET /v1/forge/balance` | اقتصاد | موجودی CU، اعتبار |
| `GET /v1/forge/pricing` | اقتصاد | قیمت بازار، تخمین هزینه |
| `GET /status` | اقتصاد | قیمت بازار، آمار شبکه، معاملات اخیر |
| `GET /topology` | استنتاج | مانیفست مدل، همتایان، برنامه شاردینگ |
| `GET /settlement` | اقتصاد | سابقه معاملات قابل خروجی |
| `GET /health` | استنتاج | بررسی سلامت پایه |

## جریان داده (Data Flow)

### استنتاج با حسابداری CU

```
مصرف‌کننده درخواست را ارسال می‌کند
    ↓
API درخواست POST /v1/chat/completions را دریافت می‌کند
    ↓
Ledger بررسی می‌کند: آیا مصرف‌کننده می‌تواند هزینه تخمینی را بپردازد؟ (can_afford)?
    ↓ بله
لایه استنتاج اجرا می‌شود (llama-server / rpc-server)
    ↓
توکن‌ها به صورت استریم به مصرف‌کننده بازمی‌گردند
    ↓
Ledger معامله را ثبت می‌کند:
  - provider.contributed += cu_cost
  - consumer.consumed += cu_cost
  - trade_log.push(TradeRecord)
    ↓
پاسخ شامل x_forge است: { cu_cost, effective_balance }
```

### خروجی تسویه

```
اپراتور اجرا می‌کند: forge settle --hours 24
    ↓
API لاگ معاملات را برای بازه زمانی مشخص می‌خواند
    ↓
محاسبه مجموع برای هر نود: ناخالص کسب شده، ناخالص خرج شده، خالص CU
    ↓
خروجی گرفتن از صورت‌حساب JSON با قیمت مرجع اختیاری
    ↓
اپراتور از آداپتور پل برای تبدیل خالص CU به BTC/فیات استفاده می‌کند
```

## مدل امنیتی

```
لایه ۰: شبکه اصلی بیت‌کوین      ← لنگر انداختن اختیاری (در آینده)
لایه ۱: امضاهای دوگانه         ← ارائه‌دهنده + مصرف‌کننده هر معامله را امضا می‌کنند
لایه ۲: دفتر کل HMAC-SHA256   ← محافظت از یکپارچگی محلی
لایه ۳: iroh (QUIC + Noise)   ← رمزنگاری انتقال
لایه ۴: اجرای استنتاج           ← مدل به صورت محلی روی ارائه‌دهنده اجرا می‌شود
```

هر لایه در برابر تهدیدات متفاوتی محافظت می‌کند:
- لایه ۴: یکپارچگی مدل (تایید هش GGUF)
- لایه ۳: محرمانگی انتقال (شنود)
- لایه ۲: دستکاری محلی (تغییر فایل)
- لایه ۱: تقلب در شبکه (ادعاهای دروغین CU)
- لایه ۰: تغییرناپذیری تاریخی (لنگر انداختن اختیاری به بیت‌کوین)

## وابستگی‌های کریت‌ها (Crate Dependencies)

```
forge-core ← انواع داده مشترک (NodeId, CU, Config)
    ↑
forge-ledger ← موتور اقتصادی (معاملات، قیمت‌گذاری، سوددهی)
    ↑
forge-lightning ← پل خارجی (کیف پول LDK، تبدیل CU↔sats)
    ↑
forge-node ← ارکستراتور (API HTTP، پایپ‌لاین، یکپارچگی دفتر کل)
    ↑
forge-cli ← رابط کاربری متنی مرجع (chat, seed, worker, settle)

forge-net ← انتقال P2P (iroh, QUIC, Noise, mDNS)
forge-proto ← پیام‌های شبکه (bincode, ۱۴ نوع محتوا)
forge-infer ← موتور استنتاج (llama.cpp, GGUF loader)
forge-shard ← برنامه‌ریز توپولوژی (تخصیص لایه، بازتعادل)
```
