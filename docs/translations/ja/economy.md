# Forge — 経済モデル

## 計算本位制 (Compute Standard)

すべての通貨システムは希少性に裏打ちされています：

| 時代 | 基準 | 裏打ち |
|-----|----------|---------|
| 古代 | 金/銀 | 地質学的な希少性 |
| 1944–1971 | ブレトン・ウッズ | 金に固定された米ドル |
| 1971–現在 | ペトロダラー | 石油需要 + 軍事力 |
| 2009–現在 | Bitcoin | SHA-256 に費やされた電力 |
| **Forge** | **計算本位制** | **有益な推論に費やされた電力** |

Forge は計算本位制を導入します。価値の単位は、有益な計算を実行するために費やされた実際のエネルギー量によって裏打ちされます。Bitcoin の Proof of Work とは異なり、Forge で費やされるすべてのジュール（J）は本物の知能を生み出します。

## CU: ネイティブ通貨

### CU とは何か

**1 CU = 検証された 10 億回の浮動小数点演算 (FLOPs) 相当の推論作業。**

CU は暗号資産（仮想通貨）ではありません。ブロックチェーン上のトークンでもありません。CU は実行された実際の計算量を表す会計単位です。CU が価値を持つのは、それが将来の計算に対する請求権であるためです。推論を提供して CU を獲得した場合、その CU を使って推論を受け取ることができます。

### なぜ Bitcoin ではなく CU なのか

| 特性 | CU | Bitcoin |
|----------|-----|---------|
| **価値の裏打ち** | 有用な計算 (固有の価値) | ハッシュ計算 (人工的な価値) |
| **決済速度** | 即時 (ローカル台帳) | 数秒〜数分 (Lightning/チェーン) |
| **取引コスト** | ゼロ | チャネル手数料、オンチェーン手数料 |
| **外部依存性** | なし | Bitcoin ネットワークの健全性 |
| **量子リスク** | なし (暗号パズルなし) | SHA-256 / ECDSA に脆弱性 |
| **利回りの発生** | あり (アイドルハードウェアで稼げる) | なし (ウォレット内の BTC は増えない) |

CU は**主要な**決済単位です。Bitcoin、ステーブルコイン、および法定通貨は、プロトコル外のブリッジアダプターを介して利用可能なオプションの**オフランプ（出口）**です。

### 生産的資産としての CU

```
賃貸アパート                    Forge 上の Mac Mini
───────────────────         ──────────────────
資産: 建物                    資産: 計算ハードウェア
コスト: 維持費                コスト: 電気代
収益: 家賃                    収益: 推論による CU
利回り: 家賃 - 維持費         利回り: 獲得 CU - 電気代コスト
空室 = 収入の喪失             アイドル = 潜在能力の浪費
```

Forge 上のコンピューティングデバイスは、ウォレット内の Bitcoin（静的な価値、利回りなし）とは異なります。それは有用な仕事を通じて収入を生み出す賃貸物件のようなものです。

## 取引モデル

### 取引の実行

すべての推論は、二者間の取引を生み出します：

```rust
pub struct TradeRecord {
    pub provider: NodeId,       // 推論を実行した人
    pub consumer: NodeId,       // リクエストした人
    pub cu_amount: u64,         // 転送された CU 量
    pub tokens_processed: u64,  // 実行された作業量
    pub timestamp: u64,
    pub model_id: String,
}
```

取引は両当事者によって記録されます。現在の実装では、各ノードがローカル台帳を維持しています。ターゲット実装では、二重署名とゴシップ同期が追加されます。

### 動的な価格設定

CU 価格は現地の需要と供給に基づいて変動します：

```
有効価格 = 基本 CU/トークン × 需要係数 / 供給係数
```

- **アイドルノードが多い** → 供給係数が上昇 → 価格が下落
- **推論リクエストが多い** → 需要係数が上昇 → 価格が上昇
- 各ノードは独自の市場環境を観察します。グローバルなオーダーブックは存在しません。

### フリーティア

貢献履歴のない新規ノードは 1,000 CU を受け取ります。これにより、誰でもすぐにネットワークを利用できます。フリーティアは最初のレクストから消費され、リセットされません。

Sybil 攻撃対策：貢献なしの不明なノード（貢献量 = 0, 消費量 > 0）が 100 を超えた場合、新規のフリーティアリクエストは拒否されます。

## 有益な仕事の証明 (Proof of Useful Work)

### コンセプト

Bitcoin の Proof of Work: 「SHA-256 ハッシュを計算するために電力を消費した。これがそれを証明するナンスである。」

Forge の Proof of Useful Work: 「LLM 推論を実行するために電力を消費した。これがレスポンスであり、それを受け取ったことを確認する消費者の署名である。」

決定的な違い：Bitcoin の証明は自己生成可能です（どのマイナーでも有効なハッシュを作成できる）。Forge の証明には、実際に推論を必要とした**カウンターパーティ（取引相手）**が必要です。需要を偽造することはできません。

### 検証プロトコル (開発目標)

```
1. 消費者がプロバイダーに InferenceRequest を送信
2. プロバイダーが推論を実行し、トークンをストリーム返送
3. 消費者がトークンを受け取り、レスポンスのハッシュを計算
4. 両当事者が TradeRecord に署名：
   - プロバイダー署名: 「私がこれを計算した」
   - 消費者署名: 「私はこれを受け取った」
5. 二重署名された TradeRecord がネットワークにゴシップ同期される
6. どのノードでも両方の署名を検証可能
```

ノードは、協力的なカウンターパーティなしに CU 残高を増やすことはできません。結託は可能ですが、経済的に合理的ではありません。結託した消費者は、虚偽の取引に署名しても何も得られないからです。

### 現在の実装

現在のリファレンス実装では、HMAC-SHA256 の整合性保護を備えたローカル台帳を使用しています。二重署名とゴシップは次のステップです。

## 利回りと評判

### 利回り

オンラインを維持し、計算資源を貢献したノードは利回りを得られます：

```
利回り CU = 貢献 CU × 0.001 × 評判 × 稼働時間 (時間)
```

評判が 1.0 の場合、10,000 CU を貢献したノードは 8 時間の夜間で 80 CU を稼ぎます。これはインフレではなく、可用性に対する報酬です。信頼性が高くオンラインであるノードは、ネットワークにとってより価値があります。

### 評判

各ノードは 0.0 から 1.0 の間の評判スコアを持ちます：

- 新規ノードは 0.5 から開始
- 稼働時間と成功した取引により評判が上昇
- 切断や失敗した取引により評判が下落
- 高い評判 → 高い利回り率、スケジューリングの優先順位

## 決済と外部ブリッジ

### 基本原則

**プロトコルは CU で決済されます。** それ以外への変換は統合側の関心事です。

### 決済明細

オペレーターは、任意の期間の監査可能な取引履歴をエクスポートできます：

```
forge settle --hours 24 --price 0.05 --out settlement.json
```

明細には、総獲得 CU、総消費 CU、純 CU、取引数、および 1 CU あたりのオプションの参照価格が含まれます。

### ブリッジアーキテクチャ

```
レイヤー 0: Forge プロトコル
  → CU 会計、取引、価格設定

レイヤー 1: 決済明細
  → エクスポート可能な取引履歴
  → 参照交換レート

レイヤー 2: 外部ブリッジ (オプション)
  → CU ↔ BTC (Lightning)
  → CU ↔ ステーブルコイン
  → CU ↔ 法定通貨
```

ブリッジ層はプロトコルの外部にあります。オペレーターごとに異なるブリッジを使用できます。プロトコルは、外部の流動性がゼロであっても有用性を維持します。

### Lightning ブリッジ

Bitcoin での決済を希望するオペレーター向け：

```bash
forge settle --hours 24 --pay
```

これにより、純獲得 CU に対して、設定された交換レート（デフォルト: 1 CU あたり 10 msats）で変換された BOLT11 Lightning インボイスが作成されます。

## エージェント主導の予算管理

### ビジョン

従来: 人間が決定 → 人間が支払う → AI が実行
Forge: ポリシーがエージェントを許可 → エージェントが予算を確認 → エージェントが CU を支出 → エージェントが実行

### API

```
GET /v1/forge/balance   → CU 残高、貢献、消費、評判
GET /v1/forge/pricing   → 市場価格、100/1000 トークンあたりのコスト見積もり
```

エージェントができること：
1. リクエスト前に残高を確認する。
2. 現在の市場価格で推論コストを見積もる。
3. リクエストが CU コストに見合うかどうかを判断する。
4. 自動的に実行し、支払う。

人間の管理者が予算ポリシーを設定し、エージェントはその制限内で自律的に活動します。

### 自己強化ループ

```
エージェント (スマホ, 小規模モデル)
  → アイドル計算資源を貸し出して CU を稼ぐ
  → その CU を使って大規模モデルへのアクセスを購入
  → より賢くなる
  → より優れた経済的決定を下す
  → さらに CU を稼ぐ
  → さらに大規模なモデルへアクセス
  → ...
```

これは応用パターンの一例です。プロトコルは市場を提供し、エージェントは戦略を提供します。

## なぜ Web3 ではないのか

ほとんどの Web3 プロジェクトは、豊富なデジタル財の上に人工的な希少性（トークン）を作り出します。Forge はその逆を行います：

- **計算資源は実際に希少である** — 本物の電力、本物のシリコン、本物の時間を必要とする。
- **CU は投機的ではない** — 未来の普及への賭けではなく、検証された作業を表す。
- **ICO なし、トークンセールなし、ガバナンストークンなし** — CU は働くことで得られる。
- **ブロックチェーン不要** — 二者間の署名とゴシップで十分である。
- **スマートコントラクトなし** — プロトコル自体が契約である。

価値はコンセンサスではなく、物理現象（電力消費）によって製造されます。
