# Forge — 脅威モデル

## セキュリティ目標

1. **転送時の機密性**: 受動的な監視者やリレーがプロンプトやレスポンスを読み取れないようにする。
2. **ピアの認証**: すべての直接接続を暗号化されたノード ID に関連付ける。
3. **限定的な信頼**: どのノードが平文を参照し、どのノードが中間状態のみを参照するかを明確にする。
4. **可用性**: 単一ピアの障害が、サイレントなデータ汚染ではなく、サービスの段階的な低下として現れるようにする。

## 暗号化

### トランスポート層
- すべての接続には QUIC と TLS 1.3 を使用 (Iroh 経由)。
- ピア認証のために追加の Noise Protocol ハンドシェイク (XX パターン) を実施。
- 結果: 前方秘匿性 (Forward Secrecy) を備えた ChaCha20-Poly1305 対称鍵暗号。
- セッションごとの一時鍵の使用 — 1つのセッションの侵害が他に影響を与えない。

### アイデンティティ (Identity)
- 各ノードは永続的な Ed25519 キーペアを保持。
- 初回起動時に生成され、プラットフォームのキーチェーンに保存。
- ノード ID = 公開鍵のハッシュ。
- 中央の認証局は存在せず、Web-of-Trust (信頼の網) モデルを採用。

### 暗号化されるデータ
| データ | 暗号化 | 備考 |
|---|---|---|
| プロンプトテキスト | はい | 転送中は暗号化。現在のリファレンスフローではシードが参照可能 |
| ストリーム出力テキスト | はい | 転送中は暗号化。生成したシードが参照可能 |
| アクティベーションテンソル | 計画中 | ランタイムで分割推論が有効になった際に関連 |
| 制御メッセージ | はい | QUIC 内のすべてのプロトコルメッセージ |
| ピアの能力情報 | はい | 暗号化されたチャネルを介して交換 |

## 脅威分析

### T1: 悪意のあるシードノード (現在のリファレンスフロー)
**脅威**: ワーカーがプロンプトテキストをシードに送信し、シードがフルモデルを実行するため、シードのオペレーターがプロンプトやレスポンスを読み取ることができる。

**現状**: これは明示的な信頼境界であり、解決済みのセキュリティプロパティではない。

**緩和策**:
- 平文プロンプトを信頼できるシードにのみワーカーを接続する。
- リレーや受動的な監視者が内容を読み取れないよう、トランスポートの暗号化を維持する。
- 中間ステージのピアが平文プロンプトを受け取らないよう、分割推論への移行を進める。

### T2: 悪意のあるパイプラインノード (ターゲットの分割推論フロー)
**脅威**: 将来のパイプライン内のノードが、中間アクティベーションからプロンプトやレスポンスを抽出しようとする。

**緩和目標**: パイプラインステージ k のノードは、レイヤー k-1 のアクティベーションテンソル出力のみを参照し、レイヤー k のアクティベーションテンソルを生成するようにする。オリジナルのプロンプトテキストは受け取らない。

**残存リスク**: アクティベーションテンソルから入力情報が漏洩する可能性がある。差分プライバシー、冗長化、およびアテステーション（証跡）は将来の課題。

### T3: Sybil 攻撃
**脅威**: 攻撃者が多数の偽ノードを作成し、パイプラインを支配しようとする。

**緩和策**:
- 観察された行動（稼働時間、正確な計算）に基づく評判システム。
- 新規ノードは低い評判から開始し、パイプライン内での配置を制限する。
- 重要なレイヤー（最初と最後）は優先的に高評判ノードに割り当てる。
- 同一 IP 範囲からの新規ノード参加に対するレート制限。

### T4: ビザンチン推論
**脅威**: 悪意のあるノードが誤ったアクティベーションテンソルを返す。

**緩和策 (MVP)**: リスクを許容する。ほとんどのユースケースでは、わずかに誤った推論結果はユーザーによって検知可能。

**緩和策 (将来)**:
- 重要なレイヤーでの冗長計算（2つのノードが同じレイヤーを計算し、比較する）。
- TEE アテステーション（Apple Silicon Secure Enclave など）を使用した検証可能な計算。
- アクティベーションテンソルの分布に基づく統計的な異常検知。

### T5: トラフィック分析
**脅威**: 監視者が暗号化された通信パターンを監視し、使用状況を推測する。

**緩和策**:
- QUIC はすべての通信を単一の接続に多重化する。
- 現在のシード/ワーカーのトラフィックでは、リクエストのタイミングやレスポンス長のメタデータが粗く漏洩する。
- 制御メッセージを一定サイズにパディングする（オプション、MVP には含まれない）。

### T6: リレーサーバーの侵害
**脅威**: ブートストラップ用のリレーサーバーが侵害される。

**影響**: 最小限。リレーサーバーは接続の確立を支援するのみ。参照可能なデータ：
- どのノード ID が接続しているか（メタデータ）。
- 暗号化された QUIC パケット（復号不可）。
- 復号されたプロンプトやレスポンスは見ることができない。

**緩和策**: 複数の独立したリレーオペレーター。DHT が構築されれば、リレーなしでネットワークは継続可能。

### T7: モデルのポイズニング
**脅威**: ノードがバックドアを仕込んだ重みを持つ、改造された GGUF モデルを提供する。

**緩和策**:
- モデルファイルを既知の良好なマニフェストに対して SHA-256 ハッシュで検証。
- モデルマニフェストはモデル発行者の署名付きで DHT を介して配布。
- ノードは検証済みのソース（HuggingFace のハッシュなど）からのみモデルをロード。

### T8: サービス拒否 (DoS)
**脅威**: ノードが参加した後に応答しなくなり、推論を妨害する。

**緩和策**:
- ハートアウトタイムアウトと再構成（リバランシング）はターゲットランタイムの特性であり、現在の実装で完全に保証されるものではない。
- ローカルへのフォールバックを、将来の分割推論クライアントの設計目標とする。
- 頻繁に切断されるノードに対する評判ペナルティ。
- 段階的な劣化（Graceful Degradation）を設計の基本原則とする。
- インバウンドの推論リクエストは、ランタイム検証とシード上の固定並列実行制限によって制限される。
- 同じピアからの重複したプロトコル `msg_id` は、制限されたリプレイウィンドウ内で破棄される。

### T9: 管理 API の露出
**脅威**: オペレーターがローカル HTTP API を保護なしで公開インターフェースにバインドし、`/status`, `/topology`, `/settlement`, `/chat` を露出させる。

**現在の実装での緩和策**:
- デーモンはデフォルトで HTTP API を `127.0.0.1` にバインド。
- オペレーターは `--bind 0.0.0.0` で意図的に公開可能。
- 公開された管理ルートは `--api-token` を介したベアラートークンで保護可能。
- JSON リクエストボディはデシリアライズ前にサイズ制限され、`/chat` でのメモリ割り当ての悪用を軽減。

**残存リスク**: ベアラートークン認証はオペレーター側の制御であり、相互 TLS ではない。トークンが漏洩した場合、交換されるまで API は侵害されたものとして扱う必要がある。

## 信頼の階層

```
最も信頼できる:  自身のデバイス (スマホ、ノート PC)
                 ↓
信頼できる:      LAN 内の自身のデバイス (自宅の Mac Mini)
                 ↓
半信半疑:        高評判の WAN ピア (数ヶ月の稼働実績)
                 ↓
信頼できない:    新規の WAN ピア (参加したばかり、履歴なし)
```

分割推論が実現した際のレイヤー割り当ては、この階層に従うべきです：
- 最初と最後のレイヤー (最も機密性が高い — 入力埋め込みと出力ロジットを参照) → 自身のデバイス。
- 中間レイヤー (中間アクティベーションのみを参照) → 半信半疑または信頼できないピアに割り当て可能。

## プライバシーの保証

**Forge が今日保証していること:**
- プロンプトとレスポンスは、直接接続されたピア間で転送時に暗号化される。
- リレーや受動的なネットワーク監視者は、復号されたプロンプトやレスポンスの内容を見ることができない。
- データパス内に必須の中央サーバーは存在しない。
- 現在のシード/ワーカーの信頼境界は明示的である。

**Forge が今日保証していないこと:**
- シードがプロンプトやレスポンスを読み取れないこと。
- 分割推論がすべてのリモート計算プロバイダーから平文を隠すこと。
- 誤ったリモート推論が自動的に検知されること。

**Forge が将来保証しようとしていること:**
- 中間ステージのピアは平文プロンプトを受け取らない。
- アクティベーションテンソルはパイプラインステージ間で転送時に暗号化される。
- プロンプトの可視性は、最小限の信頼された境界ノードに限定される。

これらの将来の保証は、実際の分割推論の提供にかかっています。それまでは、Forge は「誠実な信頼境界を備えた、暗号化されたリモート推論」として記述されるべきです。

## 経済的脅威

### T10: CU の偽造

**脅威**: ノードが TradeRecord を捏造して、稼いでいない CU を主張する。

**現在の緩和策**: HMAC-SHA256 の整合性を備えたローカル台帳により、ファイルレベルの改ざんを防止。ただし、ノードオペレーターは自身の台帳に任意の取引を書き込むことが可能。

**ターゲットの緩和策**: 二重署名プロトコル。すべての TradeRecord はプロバイダーとコンシューマーの両方によって署名されなければならない。ノードはカウンターパーティの署名なしに自身に CU を付与することはできない。ゴシップ同期により、他のノードは両方の署名を独立して検証できる。

**残存リスク**: プロバイダーとコンシューマーが結託して虚偽の取引を作成すること。これは経済的に不合理（結託したコンシューマーにはメリットがない）である。取引量や頻度の統計的な異常検知により、不審なパターンをフラグ立てできる。

### T11: フリーティアの悪用 (Sybil)

**脅威**: 攻撃者が多数の NodeId を作成し、1,000 CU のフリーティアを繰り返し悪用する。

**現在の緩和策**: 台帳内に 100 を超える不明なノード（貢献量 = 0, 消費量 > 0）が存在する場合、新規のフリーティアリクエストは拒否される。各 NodeId は Ed25519 キーペアであり、作成は容易だが追跡可能。

**ターゲットの緩和策**: ノード登録時の Proof of Work (新しい ID 作成に少額の計算コストを課す)、またはステークベースの参加 (新規ノードは消費前に計算資源を貢献しなければならない)。

### T12: 台帳の不一致

**脅威**: 異なるノードが同じ取引に対して互換性のない見解を持ち、経済的な不整合が生じる。

**現在の緩和策**: 各ノードが独自のローカルビューを維持。ノード間での整合性の保証はない。

**ターゲットの緩和策**: 二重署名された TradeRecord のゴシップ同期。両当事者は同一の署名済みレコードを作成する。ゴシップ更新を受信したノードは署名を検証し、不整合を拒否できる。Bitcoin への定期的なサマリーアンカリング (OP_RETURN) により、オプションで不変の監査証跡を提供。

### T13: 市場操作

**脅威**: ノードが需要や供給係数を人工的に膨らませて、価格を操作する。

**現在の緩和策**: 市場価格は、各ノード自身の観察からローカルに計算される。単一のノードが他のノードに自身の価格を採用させることはできない。

**ターゲットの緩和策**: 評判によって重み付けされたゴシップベースの価格シグナル。高評判ノードの観察がより重く扱われる。新規または低評判のノードは、ネットワーク全体の価格設定に大きな影響を与えることができない。

### T14: 推論品質への攻撃

**脅威**: プロバイダーが、完全な計算を行わずに CU を稼ぐために、低品質または切り捨てられた推論を返す。

**現在の緩和策**: リスクを許容する。ほとんどのユースケースでは、明らかに誤った出力はコンシューマーによって検知可能。

**ターゲットの緩和策**: コンシューマー側での品質検証。コンシューマーはトークンの小さなサンプルをローカルで再実行し、プロバイダーの出力が一致することを確認できる。スポットチェックに失敗したプロバイダーには評判ペナルティを課す。
